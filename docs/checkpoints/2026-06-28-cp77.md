**The feature.** True bouncer semantics — one persistent session, N attached
clients. `Grappa.Visitors.Login` Case 2 (registered visitor) now (1) checks the
password FIRST (auth gate — prove identity before any capacity/spawn work,
leaking no cap/circuit state to a wrong-password attempt), then (2) branches on
`Session.whereis({:visitor, id}, network_id)`:

- **live pid → ATTACH** (`attach_to_existing/3`): mint a fresh
  `accounts_sessions` row and return — the new client rides the running session
  via the visitor's user-rooted PubSub topics. NO preempt, NO respawn (so #116
  autojoin is not re-run, since `init/1` never fires), NO token revocation
  (other attached clients stay alive), NO capacity gate (nothing is spawned).
- **nil → preempt+respawn** (`respawn_path/4`, the prior fresh-spawn path,
  unchanged): capacity-gated, revokes stale tokens, stops nothing live, spawns.

**The attach verb already existed.** Minting a token for the same visitor is
exactly what share-token consume (`ShareTokenController.consume`) and the mode1
user-login path (`AuthController.mode1_login`) do; #117 was a routing decision,
not a new mechanism. Users already attached (mode1 only mints), so scope was
purely the visitor Case 2 path that had been doing preempt+respawn.

**Whereis BEFORE check_capacity — deliberate.** The capacity verbs gate new
session SPAWNS (`check_network_total` counts live `Session.Server`s;
`check_circuit` gates dialing a fresh upstream). Gating an attach on them would
block a returning identity whose session is already counted / proven-reachable —
contradicting the feature. Mirrors share-token consume (mints with no cap gate).

**Session key = identity = `{:visitor, visitor.id}`** (per `(nick,
network_slug)`) — derived from the row that already represents the identity, no
new account/identity noun. All four #117 issue open-questions resolved (see the
closing comment).
