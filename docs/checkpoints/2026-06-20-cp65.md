## S1 — 2026-06-20 — #31 visitor /invite carve-out (shipped, hot-deployed)

Third instance of the C3-lineage carve-out (C3 WHOIS → #62 `/away` →
#31 `/invite`). Visitors could not `/invite`: `handle_in("invite", ...)`
routed through `dispatch_ops_verb/3`, which rejects every visitor subject
with `visitor_not_allowed` before dispatch.

- **Fix** (`7d6859c`): route `/invite` through `dispatch_subject_verb/3`,
  thunk takes `subject` instead of `user`. `Session.send_invite/4` already
  accepted `t:Session.subject/0` (`is_subject/1` + `call_session/3`), so a
  visitor's INVITE goes to that visitor's own isolated `Session.Server` +
  upstream connection; the upstream IRC server stays the authority
  (issuer on-channel; op for `+i`). No-session → `no_session`, not
  `visitor_not_allowed`.
- **Spec challenged + confirmed safe**: a visitor invites only to channels
  their own session joined; grappa does zero stateful pre-work
  (`server.ex:1010` pure forward to `Client.send_invite`), and the Registry
  key `{:session, subject, network_id}` guarantees no cross-subject reach.
- **Moduledoc honesty fix**: the blanket "all ops verbs reject visitor
  sockets" was the drift source that mis-bucketed `/invite` — rewritten to
  enumerate the state-mutating set (op/deop/voice/devoice/kick/ban/unban/
  umode/mode/topic_set/topic_clear + oper/raw) and name the read-only +
  `/away` + `/invite` carve-outs. DESIGN_NOTES 2026-06-20 entry has the
  lineage + the transport-entitlement-vs-IRC-authority lesson.
- **Tests**: 3 mirroring the C3 WHOIS trio (live-session → INVITE upstream,
  no-session → `no_session`, malformed-nick → `invalid_nick`; the last
