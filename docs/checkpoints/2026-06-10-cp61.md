## What shipped (14 commits, `4d629fe..305db31`, merged + deployed)

### The cluster (spec `docs/superpowers/specs/2026-06-09-video-doc-uploads-design.md`)

- **Server**: per-type upload caps (`upload.{image,video,document}_
  per_file_cap_bytes`, 10/50/10 MiB defaults) replacing the single key,
  DML rename migration (ran on m42, cold deploy), MIME→category map in
  UploadsController (14 MIMEs: 5 image, mp4/mov/webm, pdf/txt/odt/ods/
  docx/xlsx), Wire + admin PUT with three cap fields.
- **cic**: `ImageHost`→`UploadHost` (per-category accept + reactive
  caps), `uploadCategory.ts` mirroring the server map, category-
  dispatched orchestrator (📸/🎬/📄), `videoTranscode.ts` on
  **mediabunny 1.46** — adaptive 720p/480p by bitrate budget under the
  50MB cap, 2' policy ceiling enforced via `<video>` metadata (binds
  without WebCodecs), transcode-always where supported, fallback to
  original on capability failures. mediabunny lazy-chunked (main bundle
  800→305kB). AdminSettingsTab: three cap inputs.
- **#49 fixed**: `lastAttempt` was written only after preCheck passed —
  retry re-sent the previous file after an oversize rejection. Now
  recorded unconditionally at dispatch. Two regression tests.
- **Latent bug 1 (Plug.Parsers)**: no `:length` set → 8MB multipart
  default below the advertised 10MB cap; 9MB uploads 413'd at the
  parser. Fixed scoped to `{:multipart, length: 64MB}` (top-level
  :length would have raised the JSON ceiling 8x — review catch).
- **Latent bug 2 (cap reactivity)**: embedded host's cap pre-check read
  a static literal; the old "reactive lookup" comment was a lie —
  admin-tuned caps were ignored client-side. Now genuinely reactive.
- **mediabunny `tags: {}` discovery**: the lib copies input metadata
  tags by default — the explicit empty object is load-bearing for the
  "transcode strips GPS/EXIF" guarantee. Pinned by test.
- e2e: 223/223 (10.2m) incl. new doc+video happy paths (real transcode
  on chromium, 1s fixture).

## Lessons

- `cmd | tail` under plain sh = set -e dead. Two deploy stages
  "succeeded" through real failures today.
- Verify third-party API claims against installed types BEFORE coding:
  the `tags` default and property-style `onProgress` both diverged from
  the design sketch; the verify-first step in the plan caught them.
- Deploy scripts by ABSOLUTE path (`/srv/grappa/scripts/...`): the
  session shell's cwd drifted to another checkout and the relative
  invocation ran the wrong repo's script three times.
