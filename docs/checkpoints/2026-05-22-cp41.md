### M3 — `bin/grappa` VERBS single-source-of-truth refactor

Pre-M3 `bin/grappa` enumerated verbs across five surfaces (per-verb
function defs, per-verb help defs, dispatch_help switch, dispatch
switch, help_top heredoc). Adding a new verb was five touchpoints.

Fix: single `declare -Ag VERBS` map keyed by kebab-name → tuple
`kind|target|group|description`. Generic `dispatch_boot` +
`dispatch_rpc` handlers for the conventional cases. Per-verb
`verb_<snake>` functions retained ONLY for arg-taking RPC verbs
(delete-visitor), attach (remote-shell), and debug verbs (open-db
+ shell). `dispatch()` uses `declare -F "verb_${snake}"` probe to
prefer a bespoke handler when one exists, falls back to the
generic dispatcher otherwise — adding a future arg-taking RPC verb
is ONE VERBS entry + ONE `verb_<snake>()` function (no dispatch-
table edit). Help banner generated from VERBS +
`VERB_DISPLAY_ORDER` + `GROUP_HEADERS`. Bash 4 limitation on
associative-array iteration order necessitates the explicit
ordered list; documented inline.

Bats: 24/24 pass. New regression test (`reap-visitors --extra`
exits 64) catches the symmetric mistake — arg-taking RPC verb
added without bespoke handler falls through to `dispatch_rpc`
which refuses with exit 64.

### M6 — `+SDio` floor at BEAM's 10-IO default

Pre-M6 `GRAPPA_DIRTY_SCHEDULERS` defaulted to `$(nproc)`, setting
BOTH `+SDcpu` and `+SDio`. On single-core deployments sqlite WAL
pool shares dirty-IO schedulers with file watchers and would
serialize.

Fix: floor default at BEAM's own 10-IO-scheduler default.
`nproc=1 → default_schedulers=10`; `nproc=20 → default_schedulers=20`.
`GRAPPA_DIRTY_SCHEDULERS` env var still wins if set explicitly.
