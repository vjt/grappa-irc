## S1 — repro + diagnosis (vjt direct browser observation)

vjt reported: "when messaging a non-existing user the incoming server
message no such nick/channel alow triggers a unread messages porco
dio marker. see it live now". Observed in the live `vjt-on-grappa-irc`
DM window: between operator's outbound `<grappa> test` and the inbound
401 reply `-raccooncity.azzurra.chat- No such nick/channel`, a
`── 1 unread message ──` marker rendered.

**Trace (server, client side-by-side).**

Server-side (CP13 `Session.Server.handle_numeric_with_routing/2`):
401 ERR_NOSUCHNICK → `NumericRouter.route` → `{:query, ghost}` →
persists `kind: :notice` row at `channel=ghost` with
`meta = %{numeric: 401, severity: :error}` → broadcasts via
`Wire.message_payload/1` on the per-channel topic. Correct — the
operator must see the failure inline in the query window.

Client-side: `subscribe.ts routeMessage` (line 216) treats `:notice`
as a content kind that bumps `messagesUnread` (sidebar bold badge).
`ScrollbackPane.tsx rows()` independently counts any
`server_time > getReadCursor()` row toward the in-pane unread-marker
— both signal paths see the 401 row and surface "unread."

**Domain class.** Same as BUG5b own-presence-event suppression
(subscribe.ts:181-189): a server-originated row that exists *because
of the operator's own action*. Operator already saw the action; the
unread alert is a false positive. The wire already carries the
discriminator (`meta.numeric` set iff the row was produced by
`handle_numeric_with_routing/2`) — adding parallel client state would
violate "don't duplicate state — derive it."

## S2 — design alignment (vjt 1-question check)

Asked vjt: "all numeric notices, or just :error severity?" — vjt:
"yes all". Both error numerics (4xx/5xx, like 401 / 482 / 404) and
info numerics (305/306 RPL_(UN)AWAY, 481, etc.) are operator-action
feedback. Predicate gates on `typeof meta.numeric === "number"`,
not on severity.

## Lessons

1. **Wire-shape carries enough context.** `meta.numeric` was already
   on the broadcast event from CP13; the cic side just had to read
   it. No protocol change needed. When investigating a UX bug,
   check what the wire already says before designing new state.
2. **Two paths, one predicate.** Sidebar badge unread + in-pane
   marker unread are independent code paths but represent the same
   semantic — "is this row unread?" Sharing the predicate (not the
   data model) keeps them aligned by construction. Future kinds
   (e.g. labeled-response routing) extend one file.
3. **Bug class, not bug example.** The CP13 design pattern (kind ↦
   meta-shape per-kind table in `Grappa.Scrollback.Meta`) gave the
   class boundary for free: "rows produced by route-to-window scan."
   Today only NumericRouter writes that shape; future producers
   would inherit the predicate.
4. **Live e2e meltdown ≠ regression.** 17 specs timing out at 30.6s
   AFTER the cp15-b6-parked spec (which itself takes 30s and
   exercises /disconnect+/connect) is a known testnet-state issue —
   the parked-spec's afterEach reconnect-poll (DESIGN_NOTES
   2026-05-09) doesn't fully cover the cascade. Targeted re-runs of
   m1/m2/m3 in isolation passed; my fix is in unread-marker code
   which none of the failed specs exercise.
