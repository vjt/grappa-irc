## Why P-0f mid-cluster

P-0e (commit `0b13a06`) routed `invite_ack` on
`Topic.channel(subject, slug, channel)` where `channel` is the
TARGET. cic only subscribes to per-channel topics for channels the
operator has joined; operators usually invite peers to channels they
are NOT in (e.g. `/invite grappa #it-opers` from #bofh). Result: the
broadcast landed on a topic with zero listeners → silent drop.

Confirmed via vjt's live browser smoke during cluster close
(`/invite Lisko #it-opers` returned no UI feedback anywhere). P-0f
flips the route to `Topic.user/1` and moves the cic mount to the
$server window. Wire payload shape unchanged; the `channel` field
becomes informational ("→ invited <peer> to <channel>") instead of
a routing key.

`feedback_silent_retry_anti_pattern` shape — the feature was
silently doing nothing in its most common usage. Caught at cluster
close by per-bucket browser smoke that
`feedback_per_bucket_deploy` requires.

## Bugs surfaced (NOT in P-0 scope, deferred to next cluster)

- **Inbound INVITE silent drop** — `:vjt INVITE grappa :#chan` from
  upstream is dropped by EventRouter fallthrough. P-0e/P-0f addressed
  the WRONG direction (operator-issued 341 ack, not inbound INVITE
  command). vjt: "I am fucking vjt on irssi inviting grappa, it's
  not fucking grappa that invites".
- **EventRouter fallthrough is silent-drop class** — KILL, WALLOPS,
  GLOBOPS, ERROR, CHGHOST, AUTHENTICATE, vendor verbs all silently
  dropped. vjt: "I do not want fucking silent drop I want to see all
  IRC messages as we have to handle them all".
- **`compose.ts` requireChannel bug** — `/invite <nick> <#chan>`
  from $server / DM / mentions silently errored with swallowed
  inline error because `requireChannel("invite")` was called even
  when `cmd.channel` was supplied. Fix on disk in worktree
  (uncommitted), rolling forward to next cluster.

## Engineering lessons

- **312 RPL_WHOISSERVER is dual-purpose**. Bahamut reuses 312 for
  WHOWAS (carrying ctime(logoff_time) in trailing) AND for WHOIS
  (carrying serverinfo). EventRouter conflict-gates: whois_pending
  takes precedence; else fold into most-recent WHOWAS entry; else
  no-op.
- **WHOWAS entries stored REVERSED** (head = most recent 314). Lets
  312 fold-into-most-recent be O(1) head-prepend instead of O(n)
  `++ [entry]` (which Credo's MapInto check rejects). Wire builder
  reads `hd(entries)`.
- **MVP scope: WHOWAS multi-history rendering deferred.** RFC allows
  N historical entries per nick; accumulator preserves all but wire
  projects only most-recent. Multi-entry rendering is a separate
  cluster if needed.
- **Cards UX renegotiation flagged**. vjt: "I am not convinced on
  cards but we can renegotiate this at a later stage". Cards stayed
  in this cluster (consistency with WhoisCard precedent +
  `feedback_card_vs_scrollback_ux`).
- **Stale orchestrator brief**: P-3 "jitter" reference in Phase 5
  list is ALREADY DONE (T31 shipped per-session ±25% jitter in
  `Grappa.Session.Backoff`). Verify before scheduling.

## Deploy

Cold-deploy at cluster close because P-0c added `whowas_pending` and
P-0d added `lusers_pending` to `Session.Server` long-lived state.
`feedback_deploy_sh_preflight_field_addition_gap`: the preflight
regex `^[+-]\s*(@type t ::\s*%\{|defstruct)` only matches the
marker line, NOT field-additions inside an existing block. Hot
deploy WOULD have crash-looped on the next message exposing the
missing field.
