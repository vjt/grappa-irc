# BUGHUNT-3 sub-cluster B (bahamut cascade) — PARTIAL CLOSURE 2026-05-25

`origin/main` `4cb0c40` — prod-side fix to `cicchetto/src/ScrollbackPane.tsx`.

## What "bahamut state pollution" actually was

NOT bahamut testnet cumulative state. NOT `join_failed` WS race.
A scroll-position race in `ScrollbackPane.tsx`:

- `applyMeEnvelope` (cold-load cursor hydrate from `/me`) is
  async via the `me()` createResource in `networks.ts:47-59`.
- `loadInitialScrollback` (per-channel REST fetch) is also async.
- When scrollback resolves BEFORE the cursor signal updates,
  `rows()` memo computes `unreadCount = 0` → no marker → the
  `scrollToActivation` rAF×2 takes the no-marker branch and
  snaps to tail with `lastElementChild.scrollIntoView({block:"end"})`.
- Cursor hydrates LATER → `rows()` re-runs → marker injects in
  the row list → marker DOM appears.
- BUT: key-effect doesn't re-fire (key unchanged) AND length-effect
  doesn't re-fire (it tracked `messages().length` which is unchanged
  — the marker lives INSIDE the rows() memo, not the messages array).
- Net result: marker DOM present, scroll glued to tail.

Symptom rotated per run because the `me` vs scrollback REST
ordering is non-deterministic and sensitive to upstream test
load. The race always existed but `bughunt-1-b-archive-mobile-seed`
(PART+JOIN cycle on `#bofh` in afterEach) perturbs the cold-load
timing reliably enough to make the bug visible from spec position
14 onwards. CP48's `restoreReadCursorToTail` fix cleared the
cursor sub-cluster but the rotating cascade family kept hitting
because the underlying scroll race was untouched.

## Fix

`ScrollbackPane.tsx:1284` — track `rows().length` instead of
`messages().length`:

```diff
   createEffect(
     on(
-      () => messages()?.length ?? 0,
+      () => rows()?.length ?? 0,
       () => {
```

When the cursor hydrates and the marker injects, `rows().length`
grows by 1 → the length-effect fires → existing `!markerScrolled()
&& marker` branch runs → `scrollIntoView(marker, {block: "center"})`.
Idempotent: `markerScrolled` latch + atBottom gate keep
subsequent appends on their existing paths.

## Carry-forward memory candidates

- `feedback_solid_memo_length_tracking`: when an effect tracks
  `signal().length` to fire on row append, the signal must be
  the same one whose RENDER drives the DOM. Tracking the upstream
  source (messages) misses commits triggered by downstream-only
  memo updates (rows = messages + cursor → marker injection).
  Trace from DOM back through memos to find the true reactive
  source for the effect's intent.
- `feedback_prod_race_disguised_as_cascade`: a rotating cascade
  across `(spec_a, spec_b, spec_c, ...)` with different victims
  per run that all share a common UI surface (here: ScrollbackPane
  initial-mount scroll) is more likely a prod-side race surfaced
  by varying test ordering than test-side pollution. Test-side
  fixes will be insufficient and rotate the failure to a different
  victim. Inspect what's COMMON about the UI surface those specs
  exercise.
