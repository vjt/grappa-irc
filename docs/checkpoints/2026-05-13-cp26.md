# CP26 — message-replay-on-reconnect cluster

Opened 2026-05-12 ~22:50 UTC after CP25 (test-infra shared-singleton)
closed clean. Cluster fixed an architectural gap, not just the
regression that surfaced it: server-side PubSub broadcasts are
fire-and-forget, so any cic session whose WS drops at the wrong
instant silently loses in-flight messages. Scrollback DB is the
source of truth; the live stream is best-effort. Cluster adds a
client-side resume mechanism + a defensive resync.

## Buckets shipped

### Bucket A — server `fetch_after/6` + REST `?after=<id>`

Commit `7d7d47c`. Mirror-symmetric to existing `fetch/6` but with
`id > after_id` cursor (NOT `server_time`). Three reasons:
  - The wire shape (`Wire.to_json/1`) already exposes `id` so cic has
    the value cheap.
  - `id` is monotonic per-row; same-millisecond `server_time` ties
    (the existing `fetch/6` docstring's caveat) become a non-issue.
  - Returns ASC for natural chronological append on the cic side.

REST controller: new `?after=<id>` param; mutually exclusive with
`?before=` (silent precedence would mask client bugs). Both unparseable
and both supplied together → 400.

Tests: 9 new in `scrollback_test.exs` (happy path, after_id newer than
newest, after_id of deleted/non-existent row, per-user iso, max-page
clamp, non-positive limit raise, network-preload contract, DM
bidirectional, own-nick narrowing, 5-arity wrapper). 5 new in
`messages_controller_test.exs` (cursor happy path, huge cursor empty,
parse error, mutual exclusion, limit cap).

### Bucket B — cic reconnect-backfill module + wiring

Commit `2eff960`. Three concerns in `cicchetto/src/lib/reconnectBackfill.ts`:

  - `recordSeen(key, msg)` — high-water mark per topic, monotonic
    (tolerates out-of-order arrivals from live + backfilled mixes).
    Wired into `routeMessage` so EVERY rendered row updates the
    cursor. Live and backfilled go through the same site by design.
  - `noteJoinOk(slug, name)` — per-topic join counter. First call
    returns false (initial subscribe, nothing to backfill); every
    subsequent call returns true (re-join after disconnect, fire
    backfill). phoenix.js's `Push.resend()` does not clear the
    `recHooks` list, so a single `.receive("ok", cb)` registered at
    first join keeps firing on every auto-rejoin — fits naturally
    on the WS reconnect lifecycle without standing up a parallel
    disconnect-detector.
  - `runBackfill(slug, name)` — REST GET `?after=<lastSeenId>`,
    dispatches each row through `appendToScrollback` (the SAME verb
    the live WS handler uses → dedupe-by-id is automatic, ordering
    preserved by monotonic id, no special interleave logic).
    Concurrency-guarded; errors log + leave cursor for next reconnect
    to retry.

`socket.ts:joinChannel` gained an optional `onJoinOk` callback parameter;
all 5 join sites in `subscribe.ts` (channels, pending, query, dm-listener,
$server) pass `() => noteJoinOk(...) && runBackfill(...)`. The DM-listener
variant carries an extra paragraph explaining why own-nick-topic backfill
recovers self-msgs only (CP14-B3 own-nick narrowing).

`api.ts` adds `listMessagesAfter` — sibling to existing `listMessages`.

Tests: 9 new in `reconnectBackfill.test.ts`. `subscribe.test.ts`'s 22
existing `joinChannel` assertions migrated to accept the new 4th
positional callback via `expect.any(Function)`.

### Bucket D — defensive resync on socket-open transitions

Commit `b7b7d01`. New createEffect in `subscribe.ts` watches
`socketHealth().state` and on every transition into "open" (post-initial)
calls `refetchNetworks()` + `refetchChannels()`. Forces the channels-loop
createEffect to re-run with fresh server-side truth so any topic added
during the disconnect window is picked up.

Closes the gap class where:

  1. Cic boots, loads channelsBySlug = [#a, #b].
  2. Operator joins #c via slash command from another grappa session OR
     a fresh tab; server broadcasts `channels_changed` on the user-topic
     — but cic's WS is mid-reconnect and the broadcast is best-effort
     fan-out (no live subscriber).
  3. Cic reconnects; phoenix.js auto-rejoins user-topic + already-known
     channel topics. #c is NOT in cic's `channelsBySlug` snapshot, so
     the channels-loop never iterates over #c, never joins its topic,
     never receives any of #c's traffic.
  4. Operator sends from #c; cic shows nothing.

Cost: 1 HTTP request per reconnect for networks + 1 per reconnect per
network for channels. Inexpensive vs the silent-empty-window class.

Together with bucket B's per-topic backfill, the cluster covers BOTH
the missed-message gap and the missed-topic gap. Per the cluster prompt:
"the right behavior is BOTH: 1. Resub ALL channels on reconnect (no
missed topic) 2. Backfill any messages that arrived during the gap."
