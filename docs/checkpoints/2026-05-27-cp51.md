### Log routing under runtime/

After two passes, the final on-disk layout in prod:

```
runtime/
├── log/erlang.log.*  ← run_erl tee of BEAM stdout (RELEASE_TMP=runtime)
├── pipe/erlang.pipe.1.{r,w}  ← bin/grappa remote attach pipe
├── grappa_prod.db (+ -shm + -wal)
├── uploads/
├── bun-cache/
└── cicchetto-dist/
```

Single source of on-disk log: `run_erl`'s tee in prod, Docker
json-file driver in dev. No parallel Elixir-side file sink (first
pass tried that; reverted — produced same lines in two paths).

Driven by `RELEASE_TMP=/home/grappa/grappa/runtime` exported by
`infra/freebsd/rc.d/grappa`'s `grappa_runas/1`. RELEASE_TMP one
level UP from `runtime/log` because run_erl always creates its
own `log/` and `pipe/` subdirs.

## S2 — 2026-05-27 — visitor rejoin + AdminEvents flake root cause + zombie respawn fix

### Visitor rejoin on bouncer restart (`63f1c98`)

vjt noticed visitors don't rejoin channels post-restart. Users do.
Root cause: `Visitors.list_autojoin_channels/1` read from a
`visitor_channels` table that had **never been written to** since
creation — the schema comment admitted `"writes will land when the
visitor-rejoin-on-restart cluster lands a producer"`, and that
cluster never landed. Also `Session.Server.persist_last_joined/4`
had a silent `defp persist_last_joined({:visitor, _}, _, _, _),
do: :ok` short-circuit swallowing every visitor snapshot.

Fix: schema-mirror with users. Migration `20260527123810` adds
`visitors.last_joined_channels` (JSON array, same shape as
`network_credentials.last_joined_channels`) and DROPs the unused
`visitor_channels` table. `Visitors.SessionPlan` carries the same
`last_joined_persister` closure pattern as Networks; the visitor
no-op in `persist_last_joined/4` is gone. Test fixture
`visitor_channel_fixture/2` rewritten to update the new column —
same signature, zero test-callsite churn. Deploy: cold (Bootstrap
reads the new column at boot).

### AdminEventsTest CI flake killed at the source (`3d196c7`)

CI run 26511978748 failed 10× `Grappa.AdminEventsTest`. Setup-time
drain (`wait_for_empty_session_registry!`) saw `Registry.select`
return leftover entries, called `Session.stop_session/2` for each,
but `stop_session` raced the DynamicSupervisor's restart of a
`:transient` worker whose Client just crashed on `:tcp_closed` —
`whereis → nil` between the dying pid and the new one, the drain
loop never converged.

Root cause traced to `auth_fixtures.ex`'s
`start_session_for/2` + `start_visitor_session_for/2` — they
spawned `Session.Server` workers and returned the pid with NO
teardown registered. End-of-test IRCServer death triggered
respawn-loops that outlived the test pid and poisoned the
SessionRegistry for the next singleton-lane test.

Two-pronged fix at the source:
1. `register_session_cleanup(pid)` helper —
   `ExUnit.Callbacks.on_exit` callback that calls
   `DynamicSupervisor.terminate_child(Grappa.SessionSupervisor,
   pid)`. Atomic, sync, idempotent on dead pids.
2. `AdminEventsTest` setup-time drain replaced with the canonical
   `Grappa.AdmissionStateHelpers.reset_session_supervisor/0`
   (walks `which_children/1` + `terminate_child/2`, no race).
   ~110 lines of inferior re-implementation deleted.

Stress-tested locally with `--repeat-until-failure 20` + full
`mix coveralls.json` (CI-identical path) — clean.

### Production zombie respawn fix (`4b16330`)

`bin/grappa list-sessions` showed a visitor pid alive without a
DB row. Trace: anon visitor created with nick `vjt`, the same as
a logged-in user already on the azzurra IRC server. Upstream 433
nickname-in-use → `Session.Server` crash → transient restart →
433 again → loop. vjt fired three admin `DELETE
/admin/sessions/visitor:<uuid>:1` and one `POST .../disconnect`;
each killed the live pid but the supervisor had already
scheduled the next restart from a previous crash, so a fresh pid
spawned with cached `init_opts` seconds later. Backoff hit 25
minutes (failure_count=9) but the slot was poisoned forever —
no path to clean shutdown short of a full app restart.

Same mechanism as the test-fixture fix above, now solved
architecturally. `Session.Server.init/1` consults an optional
`subject_row_present?` closure at the top of init; if it returns
`false`, init returns `:ignore` — DynamicSupervisor drops the
child permanently because `:transient` only restarts on abnormal
exit. The closure is supplied by both
`Networks.SessionPlan.build_plan/4` (calls
`Credentials.get_credential_by_ids/2`) and
`Visitors.SessionPlan.build_plan/3` (calls `Visitors.get/1`).
Boundary-clean: opaque function-refs, same pattern as
`credential_failer` + `last_joined_persister`.

Plumbing extended through the spawn chain:
- `SpawnOrchestrator.spawn/4` adds `{:ok, :ignored}` outcome
  distinct from `:already_started`.
- `Bootstrap.Result` adds `subject_row_gone` counter +
  `classify_outcome/3` arm. Logger metadata allowlist extended.
- `NetworksController` maps `:ignored` to `{:error, :not_found}`
  (likely cause: racing unbind during PATCH /connect).
- `Visitors.Login` maps to `:upstream_unreachable`.

Four new tests pin the contract (plan-closure tests +
SpawnOrchestrator stub + Bootstrap classify_outcome).

### Side incident: VAPID keys recovered

vjt reported push notifications failing. Logs: FCM 403 + Apple
Web Push 400 on every endpoint. Root cause: during bastille
migration `mix grappa.gen_vapid` generated NEW VAPID keypair
(`BKSBT...`) while existing `push_subscriptions` in the DB had
been firmed by browsers against the Docker prod keypair
(`BFslT...`). Push services reject deliveries whose VAPID JWT is
signed by a different key than subscription-creation key.

Fix: replaced `VAPID_PUBLIC_KEY` + `VAPID_PRIVATE_KEY` in
`/usr/local/etc/grappa/grappa.env` with the Docker values, clean
restart (`pkill -9 beam.smp + epmd`). 4 existing subscriptions
recovered (modulo the 1 already created against the new key,
which required a browser re-subscribe).
