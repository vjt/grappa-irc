### GH #118 — multi-file paste/drag-drop upload (`d7411d9`, `f401200`, `1c9569c`, `608671c`)

**The finding first (challenge-the-spec).** #118 ("paste & drag-and-drop image
upload") was **already shipped** — commit `8f1a76b` (2026-05-15, the image-upload
I-2 surface) wired `onPaste` + `onDrop`/`onDragOver` → the shared upload pipeline,
**six weeks before the issue was filed** (2026-06-27). Both surfaces already had
multi-category, in-flight progress, inline retry/dismiss, and the documented
auto-send model (`📸/🎬/📄/🎵 <url>`). The issue text's "splice URL into the
draft" contradicts that invariant — vjt confirmed **auto-send stays, no draft
splicing**.

**The one real gap:** every entry point uploaded the **first file only**, and the
orchestrator was single-slot per channel (a re-trigger *aborted* the in-flight
one). vjt chose a **sequential queue** (parallel multi-slot rejected as heavier).

**Shipped:** per-channel FIFO queue in `uploadOrchestrator.ts` — `triggerUploads`
(plural; `triggerUpload` kept as single-file alias), `pumpQueue`, `startUpload`,
`isActive`, reactive `(index,total)` counter (`uploadBatch` → cic `(i/N)` shown
only when total > 1), `resetUploadsForTests`. Settle semantics: success pumps;
error pauses (dismiss = skip-and-continue, retry = re-run failed file at front);
cancel stops the batch; decline-privacy cancels the batch. **Behavior change:**
re-trigger during an in-flight upload/transcode now **queues** (not abort-replace)
— first upload never lost; two existing tests updated. Per-file privacy gating
preserved. ComposeBox: paste/drop/picker → `triggerUploads`, `multiple` attr,
`(i/N)` counter. New e2e `uploads3-multifile.spec.ts` (3/3 iso green).
