**FIRST-DEPLOY HOTFIX `7b37af2` on main (post-S3 ff-merge):**
The new `feedback_per_bucket_deploy` cadence (sibling deploys at
each bucket close, not Z-only) surfaced TWO real bugs on its first
application — neither of which would have hit with the old
end-of-cluster deploy policy because they were latent in S1:

1. **`scripts/deploy.sh` ordering** — deployed the long-running
   container FIRST (Bootstrap fires immediately, queries
   `connection_state` from S1's migration), THEN ran migrations via
   `bin/grappa eval` in a 30s retry loop. Bootstrap's first DB read
   raced the migration, crash-looped the supervision tree, and the
   retry loop couldn't attach to a healthy `exec` target. Pre-S1
   the order was harmless. Fix: replace `up -d → exec eval` with
   one-shot `docker compose run --rm --no-deps grappa bin/grappa
   eval 'Grappa.Release.migrate()'` BEFORE `up -d`. The long-running
   container always boots against an up-to-date schema.

2. **Migration `20260504120000` sqlite-incompatibility** — `add
   :connection_state_changed_at, :utc_datetime, default:
   fragment("CURRENT_TIMESTAMP")`. sqlite refuses non-constant
   defaults on ALTER TABLE ADD COLUMN (only CREATE TABLE accepts

### CRITICAL HOTFIX caught during browser smoke (`423f4d5` on main)

`fix(channel): JSON-serialize query_windows_list at WS push boundary`

After C5 deploy, `push_user_snapshot/2` crashed in a ~2s loop on
every authenticated WS join with:

```
Protocol.UndefinedError: Jason.Encoder not implemented for
Grappa.QueryWindows.Window
```

Root cause: `push_query_windows_list/2` was passing the raw
`%{integer => [Window.t()]}` map straight into `push/3` — Phoenix
V2 JSON serializer tried to encode the Ecto schema struct
(including `__meta__`, unloaded assocs, full timestamps) and
exploded. **Pre-existing bug from C4** — sibling's C4 work created
the first `query_windows` row (`c4-test-nick`); C4 deploy worked
because empty-list path JSON-encoded `%{}` fine. C5 deploy + WS
reconnect = first non-empty list pushed = boom.

**Why C4 tests didn't catch it**: existing test asserted shape
(`assert_push("event", %{kind: "query_windows_list", windows: ...})`)
but Phoenix `assert_push/2` does NOT exercise the JSON encoder
boundary. Bug was latent until prod hit a non-empty row.

**Fix**: render each `%Window{}` to a plain map at the channel
boundary with the three fields cicchetto consumes (`network_id`,
`target_nick`, `opened_at` ISO-8601). NO `@derive Jason.Encoder` on
the schema — that would leak `__meta__` + assoc fields and breach
the "contexts return domain types, channels render at the boundary"
rule.

**Regression test added**: pushes `assert_push` payload through
`Jason.encode/1` so any future schema-serialization regression
fails in CI, not in prod.

Server gates post-hotfix: 1142 tests (+1) / 0 failures, format
clean, credo clean. Deployed + healthchecked + pushed.
