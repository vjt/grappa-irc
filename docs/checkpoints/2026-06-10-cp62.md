## The debug chain (each fix exposed the next)

1. **Symptom**: instant "too large" on a 1'44" video, no transcode
   phase. Transcode-failure reason was console.warn-only — invisible
   on iOS Safari, the dogfood platform. Fixed: reason rides the error
   UI (`Video processing failed (<reason>); the original is too
   large…`), `TranscodeError.unsupported` gained `detail` incl.
   mediabunny `discardedTracks` codes (`2e38f96`).
2. **With diagnostics visible**: "unreadable video metadata" → the
   `<video>` blob: duration probe was **CSP-blocked** (no `media-src`
   directive → fell back to `default-src 'self'`). Fixed: `media-src
   blob:` + `worker-src 'self' blob:` (mediabunny spawns codec
   workers from blob: URLs — would have been the next block)
   (`6f3327c`). e2e never saw it: nginx-test carries NO CSP — parity
   now todo High. SW-precached shell pins old CSP headers → bundle
   bump (`1346f96`, probe logs MediaError code — code 4 is the CSP
   signature) rotated the SW so installed PWAs refetch.
3. **With transcode working**: generic "network error" on upload →
   **host m42 nginx `client_max_body_size 4m`** (vhost
   `irc.openssl.it`, hand-managed, NOT in repo) cut every >4MiB body
   at the edge; conn reset mid-stream = XHR "network error". Fixed
   live: 4m → 160m (backup at /root/irc.openssl.it.bak-20260610 —
   NOTE: backups must NOT live in sites/, the wildcard include eats
   them → duplicate upstream). THIRD transport ceiling under the
   policy caps (Plug 8MB→128MiB `6e57a8d`, jail nginx 100m→160m,
   host 4m→160m). Transport stack now: host 160m → jail 160m →
   Plug 128MiB → admin caps (policy).
