**UX-7-B LANDED 2026-05-22 (autopilot, c2a71d7).** Failure A of the
baseline-fails cluster: `ux-z-cluster-journey.spec.ts:204` (200
ghost scrollback rows after archive DELETE + re-JOIN of #bofh). REAL
PRODUCT BUG, not spec rot.

Root cause: cic's `archive_changed` userTopic handler reloaded the
archive list via `loadArchive(slug)` but never invalidated the
in-memory `scrollbackByChannel[key]` for the deleted target. After
DELETE + re-JOIN, `subscribe.ts`'s join-ok callback fires
`refreshScrollback` with `?after=<high-water>` — past every deleted
row, so the server's empty page adds nothing and the pre-delete rows
persist in the Solid store. Any user who archives + deletes + rejoins
on the same tab sees ghost scrollback.

Fix: two-event split. Server `Wire.archive_purged_payload/2` (new) for
destructive DELETE path carries `target`; legacy `archive_changed_payload/1`
unchanged for PART arm (refresh-only). Cic side: new `purgeScrollback(key)`
in scrollback.ts (drops signal + load-once gate + load-more exhausted
latch + in-flight guard) + new `clearSeen(key)` in reconnectBackfill.ts
(drops per-key high-water mark) + userTopic.ts archive_purged arm
(purge → clearSeen → loadArchive). Multi-tab covered via Topic.user
broadcast fanout.

Subtle gate trap (first-attempt regression caught by reviewer round 1):
purgeScrollback no-op guard on `loadedChannels.has(key)` ALONE is
WRONG — auto-joined channels populate the signal store via
`subscribe.ts:joinOk → refreshScrollback → appendToScrollback`
WITHOUT touching the load-once Set. OR-gate
`(key in scrollbackByChannel() || loadedChannels.has(key))` covers
both populating paths. Regression test added at
`scrollback.test.ts:732-758` (drives appendToScrollback directly).
