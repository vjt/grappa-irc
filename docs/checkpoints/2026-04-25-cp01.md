**Law (sub-1):** the value of `:underspecs` is forcing helpers to be
honest about what they return. A `pos_integer()`-returning constant
function is a lie — either narrow the spec or compute the value.

**Law (sub-2):** "let it crash" applies to spec contracts too. When
the @spec says `pos_integer()`, runtime must enforce — guard clauses
(`when is_integer(x) and x > 0`) over silent `max(1, x)` clamps.
Silent clamping turns caller bugs into "works but not how you think,"
which is the worst kind of bug.

## S5 — 2026-04-25 — Phase 1 Task 4, Phoenix Endpoint + /healthz

First HTTP surface lands. Bandit-backed `Phoenix.Endpoint` joins the
supervision tree last, after `Repo → PubSub → Registry →
SessionSupervisor`, so the port opens **only** once everything healthz
would semantically attest to is alive. Two router scopes:

  * `/` (no pipeline) — `/healthz` returns text/plain `ok`. Skipping
    `:api` means no `Accept` negotiation; curl, wget, load balancers
    round-trip cleanly without headers.
  * `/` (`:api`) — JSON resource scope. Empty today; messages land
    here in Task 5.

`GrappaWeb.ConnCase` joins `Grappa.DataCase` as the second base case,
mirroring the Sandbox-per-test pattern so controller tests stay
`async: true`. Identical owner setup; only the return shape differs
(`{:ok, conn: ...}` vs `:ok`).

### Three plan deviations, all justified

1. **No `socket "/socket"` declaration in Endpoint.** Including a
   socket pointing at `GrappaWeb.UserSocket` (which lands in Task 6)
   would prevent boot — Phoenix verifies the module at init time.
   Endpoint moduledoc names the deferred mount.
2. **No `use GrappaWeb, :verified_routes_off` in ConnCase.** The
   `:verified_routes_off` macro doesn't exist; the plan itself called
   it a "hack." Phase 1 has no `~p"/..."` callers, so plain
   `Phoenix.ConnTest` imports cover everything. Add
   `Phoenix.VerifiedRoutes` when the first verified-route helper
   appears (probably Task 6+).
3. **No `import GrappaWeb.ConnCase`** in the using block. Plan
   included it for future helpers that don't exist yet — YAGNI.
   Re-add when the first conn helper lands in Task 5.

### Code review round-trip

Reviewer flagged five small things, all in five min:

  * `FallbackController.call/2` `@spec` was narrow (`:not_found |
    Ecto.Changeset.t()`) but moduledoc claimed it "centralises
    `{:error, term}` mapping" — moduledoc was lying. Fix: tighten
    moduledoc to "maps the **known** shapes only; unknown errors
    surface as Phoenix 500 via `FunctionClauseError`," keep spec
    narrow. Adding a catch-all would have hidden context bugs at the
    boundary, which CLAUDE.md "let it crash" forbids.
  * `traverse/1` private helper renamed to
    `format_changeset_errors/1`. The old name will collide with the
    recursive collect-or-bail `traverse` pattern CLAUDE.md prescribes
    for context-layer code (Task 5+).
  * Endpoint moduledoc had a wrong claim: "Session stays last because
    it reads cookies parsed by Plug.Parsers." `Plug.Parsers` parses
    the **body**, not cookies — cookies arrive via `Plug.Conn` core.
    Fixed to "Session runs before the router so handlers can call
    `get_session/2`."
  * `signing_salt: "rotate-me"` got an explicit Phase 5 callout in
    the moduledoc. No code path signs cookies today (no auth, healthz
    is unauthenticated), but Phase 5 must lift the salt to runtime
    config alongside `secret_key_base` so it's rotatable without a
    recompile.
  * Healthz "uses text/plain" assertion no longer pattern-matches a
    string prefix (which would accept `"text/plainfoo"`); asserts one
    content-type header that `String.starts_with?("text/plain")`.
