## Symptom history (the class to close)

The Application starts ONCE per `mix test` process, so
`Grappa.Session.Backoff`, `Grappa.Admission.NetworkCircuit`,
`Grappa.WSPresence`, and `Grappa.SessionRegistry` are application-wide
singletons shared across every test. ExUnit defaults to
`max_cases = schedulers_online * 2` (typically 16 on 8-core boxes);
`Ecto.Adapters.SQL.Sandbox` covers the Repo, but those GenServers +
ETS tables have no per-test sandbox.

5 separate CI surface points across the mega-cluster:
- bucket H `25737232628` `spawn_orchestrator_test:157`
- bucket I `25756898844` `spawn_orchestrator_test:251` (different
  line, same root)
- bucket H `BootstrapTest:413` flake series
- bucket H lifecycle/S5 6-test fan-out (parallelize-spawn_all)
- bucket Z `spawn_orchestrator_test:186`

Plus the bucket I local e2e flakes (cp13/cp15-b5/m9) suspected
shared-resource class but not confirmed.

## Path chosen — TI-1 Path B (max_cases=1)

Two paths considered:

- **Path A** — per-test supervised `Backoff`/`NetworkCircuit`/
  `WSPresence` instances + injected lookup name. Preserves async perf
  but invasive — touches 3 production GenServers (name + ETS table
  parameterization) + every test setup that hits these singletons
  (~30+ files).
- **Path B** — `config :ex_unit, max_cases: 1` in `config/test.exs`.
  1-line config, zero production code changes, ~22s → ~42s test-suite
  latency.

vjt-blessed Path B per CLAUDE.md "Lightweight over heavyweight."
The architectural cost of A is heavyweight relative to a problem that
surfaces ~once-per-mega-cluster. The perf delta (~20s) is bearable.

## Bucket TI-1 LANDED 2026-05-12 `35b12ba`

`config :ex_unit, max_cases: 1` in `config/test.exs` with full
rationale comment. Defense in depth: each singleton module
(`Grappa.Session.Backoff`, `Grappa.Admission.NetworkCircuit`,
`Grappa.WSPresence`, plus the `Grappa.SessionRegistry` child in
`lib/grappa/application.ex`) gains a `## Test isolation` moduledoc
paragraph documenting the `async: false` constraint.

The doc paragraph encodes the rule at the singleton declaration site
so future contributors don't reintroduce the class even if `max_cases`
is later relaxed for a faster lane (e.g. sandbox-only suite split).

Verified: 1561 tests / 8 doctests / 29 properties pass in 41.8s
sequential.

## Bucket TI-2 LANDED 2026-05-12 `ac76ee4`

Audit fixed all 18 bare-string `user_fixture(name:)` calls (10 in
`networks_controller_test.exs`, 7 in `channels_controller_test.exs`,
1 in `messages_controller_outbound_test.exs`) by appending
`#{System.unique_integer([:positive])}` (or `u()` where the helper
already exists).

CLAUDE.md "Total consistency or nothing" — half-suffixed fixtures
create two patterns and Claude propagates whichever is closer.
Necessary defense in depth even with TI-1's `max_cases: 1`
serialization.

Collateral fix: `messages_controller_outbound_test.exs` had two
hardcoded `Topic.channel("vjt", ...)` literals that survived because
the prior fixture name was `"vjt"`. The bug was masked by the
fixture-name match; with unique suffixes the topic-mismatch
surfaced on first run. Switched to `vjt.name` per CLAUDE.md "Use
production code in tests — never hardcode strings."
