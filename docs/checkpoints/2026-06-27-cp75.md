**The bug.** A prod visitor (`Takatalvi`, azzurra) had `expires_at` still
set + `password_encrypted` NULL despite successfully identifying to
NickServ — i.e. it never upgraded to the infinite-TTL identified tier.
Two independent gaps in the `+r`-observed commit rendezvous:
- `NSInterceptor` matched only `PRIVMSG NickServ :(IDENTIFY|GHOST|REGISTER)`.
  The visitor used **`ns id <pass>`** — the `ID` alias wasn't in the
  grammar → `:passthrough` → no `pending_auth` staged → `+r` arrived with
  nothing to commit.
- The `{:send_raw}` / cic `/quote` path called `Client.send_raw`
  directly, bypassing capture entirely.

**Phase 1 (`02958a9`, `74bc345`, `566babe`, `f835b47`).**
- `NSInterceptor` rewritten to the full **source-verified** azzurra
  identify set: `PRIVMSG NickServ[@host] :IDENTIFY|ID|SIDENTIFY|GHOST|REGISTER`,
  `NS|NICKSERV …`, bare `IDENTIFY|ID|SIDENTIFY` (ircd `m_identify`), and
  `PASS …` (`m_pass`→`m_identify`). Anchored at `^` so a channel body
  containing "identify"/"pass" can't false-capture; args group `(\S.*?)`
  so a verb-only line is `:passthrough`, never an empty/nil capture.
- All three outbound-line paths (`{:send_privmsg}`, `{:send_raw}`,
  `flush_lines/2`) funnel through ONE `stage_if_ns_identify/2` choke
  point in `Session.Server` — `NSInterceptor.intercept/1` now has exactly
  one call site. No identify form can bypass capture again.
- `+r` stays the commit trigger (NOT the "Password accettata" NOTICE):
  services set `+r` only when `sameNick`, so a foreign-nick identify
  fires the NOTICE but no `+r`. 10s `pending_auth` timer unchanged (`+r`
  is synchronous, sub-second). RECOVER/RELEASE deliberately NOT captured
  (no `+r`; the follow-up IDENTIFY commits). DESIGN_NOTES entry added.

**Phase 2 (`05ce58f`).** First-login auto-IDENTIFY at 001. A fresh
visitor is always provisioned anon, so first-time registration depended
entirely on the fragile manual-identify→`+r` race (lost when services
enforce-rename a protected nick to `Guest…`). `SessionPlan.with_login_identify/2`
threads the login-form password into a `:nickserv_identify` plan so
grappa sends its own canonical IDENTIFY at 001, on the connect-nick.
- **Non-obvious:** a plain field override is clobbered by
  `Session.Server.init/1`'s `Map.merge(opts, fresh_plan)` (DB-wins, the
  Azzurra zombie-respawn fix). The fix wraps the `refresh_plan` closure
  so the override survives the re-resolve WHILE the row is anon, then
  defers once `+r` commits. Mechanism lives in `SessionPlan` (owns its
  plan shape), Login holds only the policy (vjt's I1 call).

**Design fork — dropped the "+r marks indefinite regardless" twist.** vjt
floated: mark a session indefinite on any `+r`, even without a captured
password. Investigated → rejected (his call): it mints
`expires_at=NULL, password_encrypted=NULL` rows — indefinite but
**non-recoverable** (no stored credential for cross-device return), and
return-login routes on `password_encrypted` nil-ness so the row would hit
the anon token-gate; admin/operator listings key `identified ⟺ expires_at
IS NULL` and would mismark it. The password↔indefinite coupling is the
right invariant; the cure is capturing the password on every path, not
loosening the commit.
