**Bucket A — `bughunt-1(a)` `a320a4f`** — server-side PRIVMSG
auto-split on LINELEN budget.

* New `Grappa.IRC.LineSplit.split_privmsg_body/3` pure module.
  Budget = `linelen - byte_size("PRIVMSG <target> :\r\n")`.
  Splits on grapheme boundaries (UTF-8 safe). Preserves CTCP
  ACTION envelope on every fragment so each is a self-contained
  valid CTCP message. Exported via the `Grappa.IRC` Boundary.
* `Session.Server` gains `:linelen` state field (default 512;
  overridden by `005 RPL_ISUPPORT LINELEN=<N>`). State-shape
  change.
* `handle_persisting_send/3` iterates fragments via
  `persist_and_send_fragments/4` recursive loop: each fragment is
  its own persist_event + PubSub broadcast + Client.send_privmsg.
  HTTP reply returns the LAST fragment.
* Sentinel test forces `linelen: 80` via `:sys.replace_state`,
  asserts >=2 upstream PRIVMSGs + >=2 scrollback rows +
  byte-budget compliance + reconstructed body equality.
* Property test sweeps random UTF-8 bodies × random linelens
  80..512.

## Lessons captured

* **Plan vs production reality fires on EVERY bucket** — 4
  deviations in bucket A alone. Plan shape is right; execution
  always has surprises that the plan can't pre-load (Dialyzer
  success-types, Boundary exports list, Credo style nits,
  framework-specific tooling). Document each in commit body so
  future-Claude reading the diff doesn't reconstruct the plan +
  wonder why the impl diverged.
* **Hot-deploy preflight has a known gap on field-additions
  inside existing `@type t :: %{...}` blocks** — known per
  `feedback_deploy_sh_preflight_field_addition_gap` (CP28
  lesson). The AST oracle (`scripts/_extract_state_block.awk`)
  should catch these but didn't fire for bucket A. Worth a
  dedicated bucket to audit the oracle gate logic during
  bastille post-mortem.
* **Hot-deploy on state-shape change corrupts `_build/prod`** —
  per `feedback_hot_deploy_corrupts_build_prod`. Recovery is
  `rm -rf _build/prod` + `--force-cold`. Predictable + cheap to
  recover, but a sharp edge.
