### Bucket A — cursor-write contract rewrite (12 commits)

`a9e0c24..b5a1410` — moved cursor-write ownership from `selection.ts`
to `ScrollbackPane.tsx`; added input-event gate (`lastInputEventAtMs`
signal updated on `pointerdown`/`touchmove`/`keydown`) that conditions
the 500ms scroll-settle timer arm; rewrote leave-arm + blur-arm to
read `lastFullyVisibleRowId(listRef)` instead of store-tail; deleted
all four cursor-write call sites from `selection.ts` and the
`setCursorForWindow` helper. See DESIGN_NOTES "2026-05-24 — BUGHUNT-2"
for the per-commit roster.

### Bucket B — sentinel tests + bucket A fix-up (7 commits)

`75e7048..1159867` — three e2e sentinels pin the new contract
(`cursor-no-advance-on-open`, `cursor-advances-on-switch`,
`cursor-walks-with-scroll`), UX-8's three scroll-settle e2e
specs rewritten to use real PointerEvents (synthetic
`dispatchEvent("scroll")` no longer arms the gate post-BUGHUNT-2),
one vitest negative for the gate.

`b0` (`75e7048`) — fix-up landed BEFORE B1's commit. Bucket A's
`onPointerDown` handler didn't cover desktop mouse-wheel rotation
(W3C: `wheel` events do NOT fire a preceding `pointerdown` — the
pointerdown event fires on button press, not on wheel rotation).
Without an `onWheel` handler the input-event gate stayed null on
wheel scroll and the settle timer never armed → desktop operators
scrolling with the wheel never advanced their read cursor. The
inline comment in A1 was factually wrong (`"pointerdown covers
wheel-with-mouse-over-element"`); replaced + corrected. Caught by
B1 sentinel which sets cursor baseline via real `page.mouse.wheel`.
Touch + keyboard paths were correct already; only the desktop-wheel
case was missed.

### Lessons captured

* **Plan vs production reality fires on EVERY bucket** (recurrence
  from BUGHUNT-1 + UX-8 + every cluster prior). Five deviations
  across B1-B5 sentinels; every one documented in commit body per
  `feedback_plan_vs_production_reality`. Most material:
  (a) plan-suggested API surface didn't exist (`selectChannel(":home")`),
  (b) wrong assumption about Playwright input synthesis
  (`mouse.wheel` does not emit pointerdown — W3C),
  (c) stack-persistence across test specs invalidating
  exact-equality assertions on cursor values.
* **Sentinel-first development caught a Bucket A contract gap** —
  B1's first failed run (cursor never POSTed because wheel didn't
  arm the gate) was the canary for the missing `onWheel` handler.
  The halt criterion "sentinel FAILS at first run = REAL regression
  in bucket A" per the cluster-open brief fired exactly as intended.
  Fix landed as `bughunt-2(b0)` BEFORE the B1 commit (clean
  history: the sentinel and the fix it triggered are paired in the
  git log).
* **W3C event-model gotcha worth a memory** — wheel events are
  independent user inputs from pointer events; they fire even when
  the pointer is not pressed. Future "real user input" detection
  must enumerate the full set: pointerdown, wheel, touchmove,
  keydown. iOS Safari's pointerdown fires on touch-start but not
  always reliably; touchmove covers the gap. Worth a
  `feedback_dom_input_event_complete_set` memory if a third bucket
  trips on this.
* **Stack-persistence breaks naive cursor equality** — when KEEP_STACK
  reuses the integration testnet across BUGHUNT-2 specs (the per-spec
  spin-up is ~30s cold so we run them in batch), each spec inherits
  the prior spec's server-side cursor for the shared `vjt + #bofh`
  fixture. Forward-only gate (cic-side `setCursorIfAdvances` +
  server-side `ReadCursor.set/4`) silently drops smaller candidates.
  Robust assertion shape is `expect(cursor).toBe(max(baseline,
  candidate))`; pinning `expect(cursor).toBe(candidate)` fails when
  prior spec left a higher cursor. Affects B2 + B3.
