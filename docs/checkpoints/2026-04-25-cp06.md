### The seven decisions (all locked in conversation, recapped in the plan)

**A. Token model — opaque session ID, sliding 7d idle.**
- Rejected: long-lived bearer (no idle expiry → user wanted "active forever / idle 7d kills it")
- Rejected: short-access + long-refresh (refresh dance + WS-reconnect-every-15min UX cost; refresh-rotation-detects-theft is a real signal but marginal at single-operator scale)
- Rejected: rolling JWT with sliding 7d (revocation requires DB lookup anyway → JWT's "stateless" win evaporates; key compromise = total break; cargo-cult JWT shape doesn't fit our threat model)
- Accepted: opaque UUID = bearer token = session row PK; `last_seen_at` rate-limited UPDATE (60s threshold to skip DB write storm); `revoked_at` for true revocation; idle 7d via `now - last_seen_at > 7d` check
- Lesson worth pinning: JWT was designed for stateless cross-service auth in microservices fan-out. We're a monolith with one DB; we're not the use case. PASETO is the right answer if stateless ever IS needed.

**B. Upstream NickServ-pwd at rest — Cloak.Ecto AES-256-GCM, env key.**
- Rejected: DIY `:crypto` AES-GCM (reinvents Cloak; 150 LOC vs 80; same crypto)
- Rejected: libsodium `:enacl` (NIF baggage, paranoid overkill)
- Rejected: user-password-derived key (CRITICAL pushback from vjt: "key on disk is no key at all, why not derive from user password?") — DOES improve passive-disk-theft model BUT kills always-on bouncer (server has zero crypto capability without active user login → process restart = mass logout cascade → operational nightmare). Sacrificing the primary product feature for crypto property that doesn't even defend against active server compromise is bad tradeoff.
- Accepted: Cloak.Ecto + `GRAPPA_ENCRYPTION_KEY` env var; threat-model-honest documentation ("encryption-at-rest only; OTR in cicchetto for e2e")

**C. Users source — DB only via mix `grappa.create_user`; toml dies.**
- Rejected: env-var-based seed user (`GRAPPA_SEED_USER`+`GRAPPA_SEED_PASSWORD` at boot) — vjt pushback: "default user/pwd not better than no user + open registration; default admin = forces rename later; open registration should be operator-configurable"
- Rejected: bootstrap-token-in-logs (k3s/Vault pattern) — viable but premature; needs shellless-deploy as a real constraint
- Accepted: mix task only Phase 2; deferred operator-configurable registration modes (disabled/invite-only/open) to Phase 3+

**D. NickServ REGISTER proxy — defer to Phase 5.**
- Network-specific reply parsing (Anope vs Atheme vs custom — different success/failure NOTICE patterns)
- Low frequency (one-time per nick per network; weekly-monthly at most)
- Pulls in async-request/response correlation machinery shared with Phase 5 SETSTATUS, GHOST, RECOVER work
- Manual workaround trivial: operator runs `/msg NickServ REGISTER pass email` from any IRC client once, captures resulting password, drops into grappa via `mix grappa.bind_network`

**E. Upstream auth — per-network `auth_method` enum + state machine.**
- vjt critical clarification: "Azzurra (Bahamut) probably doesn't have SASL — I auth via NickServ in-band"
- Verified against `~/code/IRC/bahamut-azzurra` source: `include/msg.h`, `src/s_serv.c`, `src/s_user.c` — zero IRCv3 CAP/SASL framework. The `CAPAB` strings in code are server-to-server protocol negotiation (TS3, NOQUIT, SSJOIN, BURST), not IRCv3 client `CAP LS`. Different beast entirely.
- Bigger find: `s_user.c:1273-1278` — Bahamut has `if (sptr->passwd[0] && (nsptr=find_person(NICKSERV,NULL)))` block at end of `register_user()`. **Sending PASS at register triggers server-side `SIDENTIFY` to NickServ.** Poor-man's SASL — auth happens at register-time via legacy PASS field, server itself does NickServ handoff. No race, no post-001 IDENTIFY dance.
- Accepted: `auth_method` enum with five values: `auto | sasl | server_pass | nickserv_identify | none`. Operator declares per-network when running `mix grappa.bind_network`. `auto` = "if SASL advertised use SASL, else send PASS" (covers ergo/Libera + Bahamut/Azzurra in one default).
- Phase 5 hardening: post-registration `+r` umode check; if not authed, fall back to `PRIVMSG NickServ :IDENTIFY` retry (catches PASS-not-bound-to-services edge cases)

**F. Schema — irssi-shape: `networks` (logical) → `network_servers` (physical, many-per-net) → `network_credentials` (per-user).**
- vjt pushback: text PK on `networks` "feels off; integer or uuid"
- vjt pushback: "set network credential then autojoin? why not create network and pass everything?"
- vjt insight: "reuse irssi schema — server belongs to chatnet, chatnet has many servers"
- Accepted: `networks.id integer + slug unique`; `network_servers` has `priority` for multi-server failover (Phase 2 logic uses first only; Phase 5 adds round-robin); `network_credentials` (composite PK `user_id + network_id`) holds nick + realname + sasl_user + encrypted password + auth_method + auth_command_template + autojoin_channels (JSON array)
- Mix tasks: `bind_network` (single shot — creates network if missing, adds first server, binds credential, sets autojoin); `add_server` (additional servers per network); `unbind_network`; `remove_server`; `update_network_credential`

**G. Per-user iso on messages — `user_id` UUID FK + new index + topic shape change.**
- Rejected: shared scrollback with de-dup (de-dup key `(network, channel, server_time, sender, body)` is fragile under server-time latency variance)
- Accepted per-user model — vjt rationale: "if you disconnect, why should you keep on seeing scrollback if other user is there?" Each Session.Server writes its own rows from its own wire view. Disconnect = stop accumulating.
- vjt pushback on wire shape: "why include user_id in payload?" — accepted OMIT (client knows their own from `/me`; saves 36 bytes per row in scrollback pages)
- vjt pushback on PubSub topic: "why add user_id? can't it be inferred from session?" — answered honestly: Phoenix.PubSub topics are GLOBAL string namespaces, not socket-scoped; without per-user discriminator multi-user instances broadcast user A's events to user B's subscribers. Topic shape MUST encode user. Standard Phoenix-shape solution.
- Accepted topic shape break (Phase 1 → Phase 2): `grappa:user:{user}/network:{net}/channel:{chan}` — built via single-source-of-truth `Grappa.PubSub.Topic` module
