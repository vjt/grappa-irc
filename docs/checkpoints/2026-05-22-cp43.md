### Why this is Part 1 (not the whole bucket)

FLAKE-A manifest's "27 Class C specs are all bahamut load class"
induction was WRONG. Hypothesis-phase sampled 6 specs in isolation
(all passed) and induced the rest were load class too — but the
SPECIFIC specs sampled happened to be load-class-clean (push-install
et al. don't use `selectChannel(_, _, "Server")`). The remaining 20
specs span multiple distinct root causes:

1. **Fixture rot** — 6+ specs that call `selectChannel(_, _, "Server")`
   or use the constant `SERVER_WINDOW_LABEL` — fixed in c804208 (b0,
   b2, p0e, cp22-bnames, m2, ux-2-mobile-archive + cp13-server-window
   S8/S9, cp15-b6-pending-to-failed via shared label).
2. **Real product/spec bugs** — e.g. members-prefix-regression fails
   alone on fresh stack (no fixture call to "Server"; members pane
   doesn't render at all per trace snapshot).
3. **Genuine load class** — late-suite-only timeouts.

### Evidence

- Pre-fix baseline (run `26299521755`): 41 e2e fails.
- Post-fix Run #1: 37 fails (−4).
- Post-fix Run #2: 48 fails (+11; run-to-run variance dwarfs fix
  impact ±4).
- 12 specs newly fail in Run #2 that passed in Run #1 (m4, m5, m6,
  cp14-b1/b2/b3 — basic message-routing + scrollback). Suite-level
  non-determinism is significant.

**LANDED-with-two-green-runs DELIBERATELY NOT CLAIMED** per
`feedback_landed_claim_evidence` + `feedback_recurring_e2e_not_flake`:
the fixture fix is verified-correct in isolation (6 specs each <2s
post-fix, were 30.6s timeout pre-fix) but the broader suite is too
flaky to confidently call two green runs.

### Failed hypotheses (recorded so they're not retried)

1. **Session-bounce isolation** (`PATCH /networks/:slug park →
   connect` between specs) — IMPLEMENTED in worktree, REVERTED on
   evidence. Made things WORSE: b0/b2 still timed out at 30s on
   essentially-fresh stack (bounce ran clean but test body failed);
   from spec #61 onward all bounced specs cascaded to 0ms (the
   `bounceVjtSession()` helper itself broke under accumulated load);
   non-bounced specs like cp15-b6-pending-to-failed also failed
   (14.4s); u-z-cluster-journey afterEach log line emitted:
   `vjt did not re-join #bofh on bahamut-test within 30s` — autojoin
   restoration ITSELF takes >30s at suite scale, exactly what the
   bounce relies on.
2. **Per-spec channel-name uniquification** — never implemented;
   would only have addressed 14 of the 26 putative Class C specs
   anyway, and post-bounce evidence shows the load-class subset is
   smaller than the manifest claimed.
3. **Autojoin-restore as primary root cause** — vjt mandate
   "find out why join takes 30s do not work around it" — investigated:
   `Session.Server` autojoin loop is `Enum.reduce` over channels
   calling `IRC.Client.send_join` (fire-and-forget); the 30s is the
   gap between SEND and bahamut's JOIN/353/366 echo. `IRC.Client.
   do_connect/3` has `@connect_timeout_ms 30_000`, and the integration
   testnet has THROTTLE_ENABLE disabled (testnet's
   `options.h_hub:47-55`). No silver-bullet single-cause found —
   evidence pivoted instead to fixture-rot for the early specs.
