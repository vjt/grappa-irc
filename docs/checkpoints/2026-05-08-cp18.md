## Design summary

### bnd-A2 (Option B per vjt)

Three options weighed at design time: (A) pure helper function, (B)
Map-keyed `createMemo` + helper, (C) bigger refactor pushing
`network_id` resolution UP into `slashCommands.ts` dispatch. **Option B
chosen** — mirror-symmetric with cluster #13 M4 (`networkKey` /
`decodeChannelKey` extraction) + M7 (`target_kind/1` public-helper
promotion). +6 LOC of memo wiring vs Option A; O(1) lookup vs the 14×
O(n) scan (n is small in practice — wins are single-source-of-truth
and futureproofing, not perf).

Memo invalidates whenever the underlying `networks` resource updates
(post-/connect, post-/disconnect, bearer rotation), so callers see new
entries without manual cache management. Helper signatures take
`slug: string` (NOT `string | null`) because the 14 call sites all
hand a guaranteed-string from `submit(_, networkSlug: string, _)` —
per CLAUDE.md "Don't add error handling for scenarios that can't
happen," no nullable widening.

`compose.test.ts` mock fixture extended: mock `../lib/networks` now
exposes `networkBySlug` + `networkIdBySlug` derived from the existing
`mockNetworksData` fixture. The "/msg without active network returns
inline error" test stubs `networkIdBySlug` (was: `networks`) to
undefined — same contract under test (inline error when helper returns
undefined), updated to the new dispatch surface.

### scroll-on-window-switch (root cause + fix)

Root cause: the `[data-testid="scrollback"]` `<div>` is the SAME DOM
node across `selectedChannel` changes (Solid's `<Show>` in `Shell.tsx`
is non-keyed, so the element is reused, not rebuilt). Pre-fix the
length-effect at `ScrollbackPane.tsx:583` only fired when
`messages().length` changed, so re-selecting a previously-loaded
channel never re-snapped to its tail. The query window left scrollTop
= 0; switching back to the channel preserved that scrollTop on the
shared `<div>`.

Fix design discipline: the on-key effect already existed (resets
banner + markerScrolled); extend it rather than add a parallel effect
on the same key. Snap behavior branches on marker presence so the same
UX rule applies to both mount paths (initial focus + switch-back):
unreads → marker centered, no unreads → tail. The user's spec was
literal: "more or less in the middle of the screen, and if no unreads
then scroll to bottom."

The companion change — length-effect's `block: "start"` → `block: "center"`
— is the OTHER mount path: a window opened with unreads where the REST
page lands AFTER focus. Without this, switch-back centers the marker
but initial-focus pinned it to the top edge. Asymmetric UX is worse
than no fix at all.
