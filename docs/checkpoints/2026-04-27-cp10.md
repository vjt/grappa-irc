**Approach (b) of the C1 design call:** vite-plugin-pwa@1.2.0 +
workbox-window@7.4.0 in `generateSW` mode. Workbox embeds the
precache manifest (every emitted asset, hashed) into the SW bytes

SW byte content, which forces re-install + activate-time precache
eviction. Both bugs (S2 hardcoded cache name + S3 hashed assets
not pre-cached) fall to the same fix mechanically.

**Configuration choices:**

- `registerType: "autoUpdate"` — shell-only cache, stale assets
  never useful, no opt-in prompt.
- `injectRegister: false` — explicit `registerSW()` from
  `virtual:pwa-register` in main.tsx (deterministic vs.
  plugin-internal heuristic; `'auto'` would resolve to `false` here
  anyway because main.tsx imports the virtual module).
- `globPatterns: ["**/*.{js,css,html,svg,png,webmanifest,ico}"]` —
  precache the build's hashed JS/CSS + shell + manifest + icons.
- `navigateFallbackDenylist` for `/auth /me /networks /socket` —
  covers SPA-routing edge cases (e.g. a navigation-mode request to
  `/auth/oauth-redirect`); REST + WS protection is architectural,
  not denylist-driven (Workbox's NavigationRoute only matches
  `request.mode === "navigate"`).
- Manifest moved from `public/manifest.json` into `vite.config.ts`
  — single source of truth.

**Files removed:** `cicchetto/public/sw.js`,
`cicchetto/public/manifest.json` (both generated by plugin into
`dist/`).

## S3 — 2026-04-27 — C2 cluster: blocking init pair closed (HIGH S1 + S12)

CP10 codebase review HIGH irc/S1 + lifecycle/S12 closed: both
`Grappa.IRC.Client.init/1` and `Grappa.Session.Server.init/1` defer
their blocking work into `handle_continue/2`. Connect timeout pinned
to 30_000 ms explicitly on both `:gen_tcp.connect/4` and
`:ssl.connect/4` (was `:infinity` default — a SYN-dropped router
deadlocked init forever).

**Bootstrap inversion of meaning:** pre-fix, an upstream connect
refusal made `Session.start_session/3` return `{:error,
{:client_start_failed, _}}` synchronously and Bootstrap counted it
under `failed`. Post-fix, the failure is async — Bootstrap reports
`started=N failed=0` for any row that passed `Client.init/1`'s
validation; the per-Session `:transient` policy retries up to
`max_restarts: 3` and the `DynamicSupervisor` then terminates the
child permanently. Operators grep `(stop) {:connect_failed, _}`
from the Session crash trace under the new semantic. Phase 5
reconnect/backoff replaces the exhaust-and-give-up shape.

**Bonus root-cause fix:** pre-existing `Session.stop_session/2` race
(~1-in-5 flake on the existing teardown test). `DynamicSupervisor.
terminate_child/2` returns when the child is dead, but
`Grappa.SessionRegistry`'s OWN process-monitor on the dying pid runs
in the Registry process and may not have processed its DOWN yet.
`stop_session/2` now monitors the pid itself, awaits the DOWN with a
5s `Logger.error`-on-timeout budget, then polls `Registry.lookup`
until the entry is gone (5ms × 100). The race surfaced reliably
while iterating on the C2 test; chained `stop_session/2` →
`start_session/3` would otherwise occasionally fire `:already_started`
against a zombie pid.

**Test-side discipline pinned:** the Session-level non-blocking-init
test uses `Grappa.Session.Server.start_link/1` directly (linked to
the test pid), NOT `Session.start_session/3` via DynamicSupervisor.
The latter would trigger the connect-refused crash → `:transient`
restart cycle, which exhausts `SessionSupervisor`'s `max_restarts: 3`
budget in <100 ms and crashes the supervisor — torching every other
Session in the test run. Pinning the GenServer init contract
directly is the right surface; the supervisor path is the wrong
unit-of-test.
