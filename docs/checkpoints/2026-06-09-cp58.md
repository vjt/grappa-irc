### 1. cic `/msg` to a new nick — own-send cursor poison (issue #50, `dd61f10`)

vjt-adjacent: surfaced as the recurring e2e flake
`m6-cicchetto-to-priv-opens-query`, but it was a **real cic production
bug**, not a timeout. `/msg <new-nick>` opens an EMPTY query window then
sends; `sendMessage` advanced the server read cursor to the just-sent
row's own id even though no row had rendered. When the live WS broadcast
lost the per-channel subscribe race (brand-new topic), the join-ok
`refreshScrollback` recovery resolved its resume cursor via
`getResumeCursor` — which falls back to the read cursor when nothing was
ever `recordSeen`'d — i.e. to the own message's own id. `?after=<own-id>`
→ empty → row never recovered → "no messages yet" until reload.

**Fix — gate the poison at its source:** `sendMessage` advances the
read cursor only when the local pane already holds a rendered row. Empty
pane → cursor stays null → `refreshScrollback` resumes from id 0 and
recovers the send. Established channels unaffected (their high-water
mark already shadows the read cursor in `getResumeCursor`). ~5 prod
lines; `refreshScrollback`/`getResumeCursor` untouched.

**Rejected issue #50's own proposed fix** (clamp `cursor=0` inside
`refreshScrollback` when the pane is empty). The channels loop joins
EVERY channel eagerly while `loadInitialScrollback` is focus-only, so an
unfocused channel with a `/me`-hydrated read cursor R has an empty pane
when its join-ok refresh fires — the clamp would fetch `?after=0` (oldest
200) instead of `?after=R`, leaving an unreachable gap in the pane on
later focus. The spec inherited a bug (CLAUDE.md "challenge the spec").

Validation: vitest 1783 pass (RED→GREEN anti-poison test), biome+tsc+build
green, adversarial review SOUND (the reviewer reverted the gate to prove
the test isn't vacuous), full e2e green — m6 now renders the own row in
**2.1s** (was failing at 7.5s/15s). DESIGN_NOTES 2026-06-09 entry; the
prior "m6 = genuine timing" misdiagnosis corrected at its source. Issue
#50 CLOSED. The m6 15s round-trip timeout bump is kept as harmless
slow-Pi headroom, not the fix.

### 2. e2e-tooling self-heal — stop re-hitting two landmines every session

Docs already documented both (TESTING.md), and we kept eating them anyway
— so the scripts now self-heal instead of `die`-ing on a manual step:

- **`scripts/testnet.sh`** auto-inits the `cicchetto/e2e/infra`
  (azzurra-testnet) submodule on `up` — fresh git worktrees don't inherit
  submodules, so every worktree e2e used to die at the bahamut check.
- **`scripts/_lib.sh:e2e_force_rm`** — plain `rm` → non-interactive
  `sudo` for root-owned survivors. `testnet.sh up`/`down` use it for
  `runtime/e2e/{cicchetto-dist,grappa-runtime}`, which a prior run can
  leave uid-0 (despite `--user`), aborting the NEXT bring-up with
  cicchetto-dist AccessDenied / sqlite `database_open_failed` / "Pool
  overlaps". Tested against a real root-owned file. Worktree-removal
  blocked by root-owned Playwright `test-results` is documented (kept as
  artifacts; manual `sudo rm` + `git worktree prune`).

### 3. cic read-cursor POST guard (issue #44, `f22568d`, bundle `BF6Dside`)

Prod logged 31 `POST .../read-cursor` 400s on service-nick query windows
(NickServ 17 / ChanServ 10 / OperServ 4). `setReadCursor` POSTed whatever
id the caller handed it; service-nick windows settle/blur before a
positive persisted id exists, and the settle candidate (`lastFully-
VisibleRowId` → `Number.parseInt` → NaN on empty `data-msg-id`) slips
past `setCursorIfAdvances`'s forward-only gate when the window has no
cursor yet. Fix: positive-int guard as the first statement of
`setReadCursor` (`!Number.isInteger(messageId) || messageId <= 0` →
return), skipping BOTH the optimistic advance and the POST — mirrors the
server's `is_integer and > 0`, at the POST-owning boundary so every
caller inherits it. RED→GREEN (it.each 0/-1/NaN/1.5 + advance-skip +
valid-still-posts), adversarial review SOUND (mutation-verified). Issue
closed. Follow-up (Low): drop the m42 fail2ban `/read-cursor` exemption
once prod 400s hit zero.
