## The subject parity invariant

Pre-cluster, every server-side feature surface that touched "who
owns this row" branched on `{:user, _}` vs `{:visitor, _}` and
either refused the visitor branch outright (`require_user/1` plug,
`check_not_visitor/1` `with`-guard) or accepted it via a parallel
short-circuit code path (`maybe_dispatch_push(_, %{subject:
{:visitor, _}}), do: :ok`). V1-V9 collapse those branches: every
persistence-write codepath builds its changeset via
`Grappa.Subject.put_subject_id/2`, every read query goes through
`Grappa.Subject.subject_where/3`, every controller picks subject
from `Subject.from_assigns/1`.

Three subject-scoped tables (`query_windows`, `push_subscriptions`,
`user_settings`) gained the XOR FK pattern that `read_cursors` had
since CP29 — `(user_id IS NULL) <> (visitor_id IS NULL)` CHECK +
two partial UNIQUE indexes per subject branch + ON DELETE CASCADE
to both parents. V5's cascade test asserts deleting a visitor
wipes all five owned tables in one Reaper sweep — the database
does the work.

The invariant going forward: any per-subject behaviour difference
that REMAINS post-cluster must be explicitly justified. Today the
only one is V7's TTL semantics (anon vs identified expiry).

## Two-tier identity model

| Subject | Auth proof | Data lifetime |
|---------|------------|---------------|
| Anonymous visitor | none (visitor row + bearer) | 48h sliding TTL — Reaper sweep + 5-table FK CASCADE wipes everything |
| NickServ-identified visitor | NickServ password verified vs upstream `+r` MODE observation | **infinite** — `expires_at = NULL` |
| Registered user (admin/operator) | local Argon2 password (`users.password_hash`) | infinite (operator-only path via `mix grappa.create_user`) |

V7 codifies the lifetime model. Anon visitors get
`expires_at = now + 48h` on every touch; identified visitors get
`expires_at = NULL` written at the `commit_password/2` transition
(called from
`Session.Server.apply_effects([{:visitor_r_observed, password} | _], _)`
when upstream signals `+r`). Subsequent `Visitors.touch/1` calls
become no-ops for identified visitors. Reaper's sweep query carries
`WHERE expires_at < now() AND expires_at IS NOT NULL` so identified
visitors are skipped automatically.

## Why V8 dropped

Pre-cluster spec carried a V8 "promote visitor → registered user
with reparenting transaction" bucket. The 2026-05-15 spec
refinement dropped it: NickServ identification with infinite TTL
already provides everything a "registered user" tier would. The
visitor row stays a `visitor_id` row forever; capability-equality
with `users` is established by the V1 XOR FK migrations. No
double-password UX problem, zero data-migration code, zero
double-account-state classes. The "registered user" tier exists
ORTHOGONALLY as the admin/operator account path (the bouncer admin,
future read-only dashboard accounts) — not a visitor's promotion
target.

## V9 NICK rename safety analysis

Pre-V9 the visitor branch at `nick_controller.ex:61` returned
`403 forbidden`. V9 lifts the gate. Two lines of defense protect
the `(nick, network_slug)` UNIQUE on `visitors`:

1. Pre-check `Visitors.nick_in_use?(visitor_id, target_nick,
   network_slug)` BEFORE the upstream NICK frame — catches >99% of
   collision races at the controller boundary, returns 409
   `nick_in_use`.
2. UNIQUE constraint at the EventRouter persist site — second line
   of defense via `Visitors.update_nick/2`'s
   `unique_constraint(:nick, :network_slug)`. Logged + dropped on
   collision.

User subjects don't carry the persister callback — their nick lives
in `Networks.Credential` (operator-driven). Visitor subjects route
through an injected `visitor_nick_persister` function-ref (mirror
of `visitor_committer` for `+r` MODE).

vjt vetoed the orchestrator's complex sync-wait + 422-on-433-numeric
+ `pending_nick_rename` correlation field design. User path is
fire-and-forget 202 today; visitor=user per the parity invariant;
432/433 silently leaves DB unchanged via natural EventRouter shape
(no echo → no effect → no DB write); cic already listens to
`own_nick_changed` (CP-15). Pre-existing UX hole around 432/433
silence is orthogonal to V9.

## HOT-vs-COLD preflight gap (V9 incident)

`Session.Server`'s `@type t :: %{...}` got a new
`visitor_nick_persister` field — per
`feedback_deploy_sh_preflight_field_addition_gap` the AST oracle
SHOULD have caught it and demanded COLD. The AST oracle never ran.

The deploy operator did `git merge --ff-only` BEFORE invoking
`scripts/deploy.sh`. The deploy's `git pull --ff-only` returned
"Already up to date", so the preflight diff base
(`HEAD@{1}..HEAD`) was empty → AST oracle saw nothing → false HOT
classification → `Phoenix.CodeReloader` fired against a state-shape
change.

Live BEAM survived the hot reload (no immediate crash) but
`_build/prod` got corrupted per
`feedback_hot_deploy_corrupts_build_prod`. Subsequent `--force-cold`
rebuild failed `compile_env validation`. Recovery:
`rm -rf _build/prod && scripts/deploy.sh --force-cold`. ~30s
downtime; visitors auto-respawned via Bootstrap.

Captured in `feedback_deploy_preflight_empty_diff_after_merge`.
Until `scripts/deploy.sh` learns to compare against
`origin/main@{1}..origin/main` (the actual pre-pull remote state) or
persists a last-deployed-SHA marker, the operator must manually
inspect `lib/grappa/hot_reload/long_lived_modules.ex` + migrations
+ `mix.lock` post-local-merge and pass `--force-cold` defensively.

**Law:** every safety check has an implicit assumption about WHEN
it runs. `scripts/deploy.sh`'s AST oracle assumed the merge
happens INSIDE the deploy. The CLAUDE.md "merge → deploy" workflow
has the merge BEFORE. Empty diffs are not a quiet "no changes" —
they're a load-bearing signal that the diff base is wrong.

## Engineering lessons

- **One feature, three usage classes, ONE invariant.** When
  `read_cursors` shipped its XOR FK pattern in CP29, three sibling
  tables (`query_windows`, `push_subscriptions`, `user_settings`)
  carried the legacy user-only shape. CP32 collapses the
  inconsistency in one cluster. Mixed patterns drift; consistent
  patterns compound.
- **Drop, don't promote.** V8 spent zero code because the spec
  refinement caught the redundancy at brainstorm. NickServ +r
  observation IS the cryptographic identity proof; demanding a
  second password (the "promote" double-secret) was inventing UX
  problems to solve UX problems that didn't exist. Brainstorm-rigor
  saved an entire data-migration code path.
- **Veto orchestrator complexity that exceeds the existing pattern's
  shape.** V9's NICK rename came in 1/3 the size of the orchestrator
  brief proposed. The brief's sync-wait + correlation field design
  was a defensible shape — but user nick-rename is fire-and-forget
  202 today. visitor=user per parity → fire-and-forget 202 for
  visitors too. Pre-existing pre-existing UX holes around 432/433
  silence stay open as their own future cluster.
- **Push notifications shipped INSIDE the cluster that lifted their
  schema.** When the V1 migration changes `push_subscriptions` to
  the XOR FK shape, `Push.Sender` MUST be rewritten in the same
  cluster — leaving it user-only would have left the cluster
  incomplete. Folding the post-CP31 "push notifications" cluster
  into V3 cost less than running it separately would have.
