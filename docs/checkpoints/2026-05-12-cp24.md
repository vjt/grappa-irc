## Three CRITICAL findings (the porco dio trifecta — corrected to two 2026-05-12 bucket A)

- **C1 — SASL credential leak in pre-handshake phases**
  (`lib/grappa/irc/auth_fsm.ex:232-234`). `AUTHENTICATE +` clause
  matched UNCONDITIONALLY for any phase other than `:registered`.
  Hostile/MitM upstream can elicit operator's SASL credentials before
  SASL has been negotiated. Phase guard missing on `:sasl_pending`.
  ✅ **LANDED 2026-05-12 bucket A** — phase pin on `:sasl_pending` +
  catch-all absorption + 4 regression tests (3 pre-handshake phases ×
  stray AUTHENTICATE + plus 1 control proving the legitimate
  `:sasl_pending` reply still fires).
- **C2 — SQLite `PRAGMA foreign_keys` never enabled in dev/prod**
  (`config/runtime.exs:22-27` + `dev.exs:3-6`). 23 migrations'
  worth of `references(..., on_delete: ...)` are runtime-dead.
  ❌ **FALSE FINDING — corrected 2026-05-12 bucket A.** Live-container
  probe confirmed `PRAGMA foreign_keys=1` AND that orphan-FK insert
  raises `Exqlite.Error "FOREIGN KEY constraint failed"`. The
  `ecto_sqlite3` adapter sets PRAGMA foreign_keys=ON on every
  connection-init by default (see `deps/ecto_sqlite3/lib/ecto/adapters/sqlite3.ex:85`
  + `deps/exqlite/lib/exqlite/pragma.ex:52`). The reviewer read
  "SQLite ships with PRAGMA foreign_keys=OFF" as the runtime default
  and missed the adapter override. The `validate_subject_exists/1`
  pre-flight checks in `Accounts.create_session/4`, `QueryWindows.open/4`,
  `UserSettings.get_or_init/1` exist for a SEPARATE, REAL ecto_sqlite3
  limitation: the engine returns FK constraint NAME as `nil` so
  `assoc_constraint` cannot pattern-match — the pre-flight produces a
  clean changeset error before the insert raises. The existing source
  comments already describe this correctly; no source edits required.
  Persistence/S7's "without S1 fixed there is NO backstop" framing
  also wrong — re-validate at bucket B. Tally drops from 3 CRIT to
  **2 CRIT** (C1+C3); compiled review doc + persistence draft updated
  with correction sections.
- **C3 — Visitor WHOIS broken** (`lib/grappa_web/channels/grappa_channel.ex:445-454`).
  Comment explicitly flags the bug + prescribes the fix; implementation
  uses the rejected path. Visitors get `visitor_not_allowed` despite
  documented intent.
  ✅ **LANDED 2026-05-12 bucket A** — new `dispatch_subject_verb/2`
  helper + `resolve_subject/1` extracts `{:visitor, id}` from the
  `"visitor:<uuid>"` user_name; WHOIS dispatch swapped over; 3
  regression tests (live visitor session sends WHOIS upstream;
  visitor-without-session returns `no_session`; CRLF nick rejected
  as `invalid_line`).
