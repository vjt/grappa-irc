**The feature.** One mechanism covers both locks. On a `473`/`475` for a
channel in the boot autojoin set, `Session.Server` sends
`PRIVMSG ChanServ :INVITE #chan` and records it in a new per-session
`awaiting_invite` MapSet (one attempt per channel = the dedupe). If
ChanServ relays the inbound `INVITE` (only when the bouncer's identified
account holds ≥VOP on the registered channel), `EventRouter`'s
inbound-`:invite` clause emits `{:rejoin_invited, ch}` →
`Client.send_join(ch, nil)` **keyless** + `record_in_flight_join/2`
(window `:failed → :pending → :joined`). Source-verified:
`bahamut-azzurra/src/channel.c can_join (~:1919)` short-circuits on the
invited-list BEFORE the +i (:1940) and +k key (:1968) checks — the invite
beats the key, so no stored key is needed. Send wire verified against
`services/src/chanserv.c:6205/6239` (one arg, invites self).

**Scope (vjt-confirmed).** Autojoin only — manual `/join` of a +i/+k
channel is unchanged (not in `state.autojoin` → stays `:failed`, keeps
cic's `[Join]` CTA). No +k key storage; that residual niche (channels
where the bouncer has NO ChanServ access, so can't be invited) is #113,
**downgraded to low-prio** this session. Non-awaiting inbound INVITE still
flows to the `:server_event` persist path (cic CTA, b2) — the EventRouter

**Autojoin-vs-manual is derived, not flagged.** Membership in
