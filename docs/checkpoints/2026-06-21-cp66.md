## Session 2026-06-21 (4) — duplicate NickServ IDENTIFY on login (#27) ✅ SHIPPED

Mezmerize (Azzurra) reported the NickServ "risulti identificato" NOTICE
showing **twice** on registered-visitor login. Not a cic render bug —
grappa put `PRIVMSG NickServ :IDENTIFY <pw>` on the wire twice: (1)
`IRC.AuthFSM.maybe_nickserv_identify/1` at 001 RPL_WELCOME (the canonical
site — fires for every `:nickserv_identify` spawn incl. Bootstrap
crash-respawn; staged for the +r MODE observer via
`Session.Server.maybe_stage_pending_password/1`, NOT NSInterceptor, since
the AuthFSM emission bypasses the `send_privmsg` call path), and (2) a
second, redundant `Login.send_post_login_identify/3` post-readiness. A
case-2 visitor is *always* `:nickserv_identify` (no SASL path in
`Visitors.SessionPlan`), so (1) already produced the same NOTICE + +r
MODE before (2) ran.

Fix: delete the post-readiness send + its now-dead helpers
(`send_post_login_identify/3`, `error_tag/1`, `require Logger`). No new
state. Side benefit: that path was the lone site threading the cleartext
password through `Session.send_privmsg`. Strengthened `login_test` to
assert IDENTIFY appears on the wire **exactly once** via a TCP-order
barrier (push `PRIVMSG NickServ :HELP`, wait for it, then count —
`packet: :line` + `active: :once` guarantee earlier lines are buffered);
fails on old code (`got 2`), passes after.

## Session 2026-06-21 (5) — orphaned PWA icon badge (foreground reconcile) ✅ SHIPPED

vjt reported the home-screen icon badge sticking at "1 unread" after
reading everything. His instinct was "reset the badge on app open" —
pushed back: a client-only reset wouldn't hold (the `/me` seed re-stamps
on open) and a blind reset-to-0 would wipe a legit background-arrived
badge. **Debugged with data first**: prod rpc against the live node
(`Grappa.Push.BadgeCount.count/1` for the operator) returned **0** — the
server count was correct, refuting a server-side stuck-count hypothesis.
The drift was purely the OS icon-badge SURFACE.

Root cause: two writers, no shared state. The service worker push handler
(`service-worker.ts applyIconBadge`, door #1) calls `setAppBadge`
directly off-signal while backgrounded; the in-page `mountBadgeSync`
effect only re-fires on a signal *change*. A warm foreground reading
0-over-0 → no signal change → effect never re-fires → SW-set badge
orphaned. Cold launch was fine (`/me` seed + mount reconcile); warm
resume had no reconcile point.

Fix (`fix/badge-foreground-reconcile`, `f4f8012`): `mountBadgeReconcile`
adds a `visibilitychange` listener that re-pulls the authoritative `/me`
`badge_count` and `reconcileBadge` force-applies it (bypassing the
signal-equality skip). Reconciles to the SERVER count, not a blind 0, so
a genuine background-arrived badge survives. TDD: RED→GREEN unit tests in
`badge.test.ts` (force-clear with signal already 0; visibilitychange
wiring; hidden no-op; null no-op; disposer removal).
