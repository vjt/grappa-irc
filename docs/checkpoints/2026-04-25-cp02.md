## S6 — 2026-04-25 — Phase 1 Task 5, GET /messages

First read resource lands. `MessagesController.index/2` exposes
`Grappa.Scrollback.fetch/4` over HTTP at
`/networks/:network_id/channels/:channel_id/messages` with
`?before=` (cursor) and `?limit=` (page size, default 50).
`MessagesJSON` view renders the flat array — using
`render(conn, :index, ...)` instead of inline `json/2` keeps the
surface aligned with Phoenix 1.8 `formats: [:json]` and gives Task 6
a place to add `:create` without a controller refactor (this was a
plan-body deviation from the plan-body's inline serialization; the
plan's own Files list mentioned `messages_json.ex`, so the body was
out of sync with itself).

### Code review surfaced a real bug

Reviewer flagged `?limit=banana` falling back silently to the default
as a CLAUDE.md violation ("validate at boundary, reject unknown
values"). Initial impl matched the plan, which permitted any int (and
had its own bug — `?limit=0` would be passed to
`Scrollback.fetch/4`, crashing on the `when limit > 0` guard, which
in turn would 500). Correct shape is to distinguish *absent* (use
default) from *present-and-unparseable* (return 400). Helpers now
return `{:ok, n}` / `{:ok, default}` / `{:error, :bad_request}`;
controller uses `with`; `FallbackController` gets a `:bad_request`
clause and the spec widens to match.

This is the second time a plan deviation flagged a plan bug rather
than a code bug — a healthy pattern. Plans aren't sacred; CLAUDE.md is.

### Reviewer was wrong about Jason atom encoding

The "MUST FIX" claim that `:privmsg` would crash Jason was based on
an incorrect premise — Jason's default behavior IS to encode any
atom as its string name. Verified at REPL:
`Jason.encode!(%{kind: :foo}) == ~s|{"kind":"foo"}|`. Tests passing
were the actual evidence (they round-trip through the Phoenix JSON
pipeline).

But the *related* point — that the contract wasn't asserted in tests
— was correct. Added `assert Enum.at(body, 0)["kind"] == "privmsg"`
to one of the tests so the contract is now load-bearing in the test
suite, not just a moduledoc claim.

Pattern: when a reviewer flags a MUST FIX based on assumed behavior,
verify the mechanism before changing code. The reviewer can be wrong
about the cause and right about the gap.

### Test coverage tightened

Added cross-channel and cross-network isolation test — would have
caught a `Scrollback.fetch/4` regression that dropped its `WHERE
channel == ^channel` clause. Previous tests would have shipped that
bug green because every seed row used the same `(network_id,
channel)` pair. This is the "if the implementation were wrong, would
this test catch it?" CLAUDE.md test-quality bar. Plus three 400-input
tests for the boundary-validation tightening.

## S7 — 2026-04-25 — Phase 1 Task 6, POST /messages + first PubSub

First write resource lands. `MessagesController.create/2` inserts a
`:privmsg` row with `sender = "<local>"`, returns 201 via
`MessagesJSON.show/1`, and broadcasts the new event over
`Phoenix.PubSub` on `grappa:network:{net}/channel:{chan}` so Task 7's
Channel handler can `push/3` the event verbatim.

### Plan deviations (CLAUDE.md > plan)

Three improvements over the plan as written:

- `kind: :privmsg` (atom), not `kind: "privmsg"` (string). Schema is
  `Ecto.Enum` over `[:privmsg, :notice, :action]`; both forms cast
  but CLAUDE.md "atoms for closed sets" picks one.
- Wire shape goes through `MessagesJSON`, not an inline
  `serialize/1`. Reused Task 5's view module by making
  `MessagesJSON.data/1` public + adding a `:show` render. REST and
  PubSub now share one wire shape per CLAUDE.md "every door."
- `:ok = Phoenix.PubSub.broadcast(...)` match-or-crash. Plan didn't
  match the `:ok | {:error, _}` return; Dialyzer flagged it.
  Match-or-crash is the let-it-crash answer — PubSub child failure
  is catastrophic, supervisor restarts, client retries.

### Code review surfaced two real tightenings

Reviewer flagged the catch-all `def create(_, _)` as too greedy: it
swallowed BOTH bad client body AND router-config drift (renamed
`:network_id` → `:net` would silently 400 instead of 500ing loudly).
Tightened to `def create(_, %{"network_id" => _, "channel_id" => _})`
— same blast radius for legit 400s, `FunctionClauseError` for path
drift.

Reviewer also pushed back on the broadcast event's duplicated outer
`body` field (paired with the nested `message: %{body: ...}` map).
Argument: ergonomic shortcut in tests, but the duplicated field
ships to every cicchetto WS client forever as two fields with the
same value and no canonical source. Dropped. Test assertions now
descend into the nested map (three extra characters); wire shape
stays single-sourced. **Generalizable lesson** captured in
project-story S7 as a Law.

- Persistence — POST → subsequent GET returns the row
- Scoping — broadcast does NOT leak to a sibling channel topic

### Code review surfaced three real tightenings

Reviewer flagged three SHOULD-FIX:

3. User-topic happy-path test was a partial-map assertion. Channel

Plus C1 (Phase 2 authz breadcrumb on user-topic clause), C2 (drop

### Cross-test PubSub pollution caught at the seam

After review fixes, suite went red on the sibling-channel scoping
test — receiving a broadcast it shouldn't. Cause: `Phoenix.PubSub`
topics are global, not per-test. Channel test joined
`grappa:network:azzurra/channel:#sniffo` while the controller test
ran async and posted to the same `(net, channel)`, broadcasting on
the same topic. Test process's channel was a real subscriber and
got the broadcast.

Fix: distinct `(network, channel)` pairs per channel test
(`ch_happy_net/#ch_happy`, `ch_sibling_net/#ch_joined`,
`ch_user_test`). Schema columns are free-form strings — trivial.
Documented in the test moduledoc so the next person doesn't repeat.
**Generalizable lesson**: any test that subscribes to PubSub by
topic name is competing for a shared resource with every other
async test. Captured as a Law in project-story S8.
