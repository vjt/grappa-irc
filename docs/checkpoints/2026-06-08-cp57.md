## What shipped (DONE + DEPLOYED to m42, hot — no BEAM restart)

### 1. Optimistic forward-only read-cursor advance (`7839244` fix, `c71c1ee`+`9a102a6` docs; bundle `DHGmyJpO`)
vjt report: (a) leaving a channel with nothing unread flashed a sidebar
badge for a frame, and (b) an own-sent message sometimes rendered above
the `── N unread ──` divider after stepping to another window and back.

One root cause: the local read-cursor signal (`readCursor.ts`) was
ROUND-TRIP-ONLY — `setReadCursor` POSTed and the local map advanced only
when the server's `read_cursor_set` echo landed. The POST→echo gap is a
stale-cursor window; the focused-window badge suppression (drops sync on
`selectedChannel` flip) and the `markerCursorId` re-latch (on focus
acquisition) both fired inside it and read the OLD cursor.

Fix: `setReadCursor` advances the local signal OPTIMISTICALLY,
forward-only, before the POST — one place, every write path inherits it.
Lands in the same Solid flush as the suppression-drop + re-latch.
Forward-only so an in-flight stale POST can't clobber a peer's newer
advance (only the unconditional `applyReadCursorSet` echo moves the
cursor backward). Freeze contract untouched (divider reads frozen
`markerCursorId`, not the live signal).

**Revises cp56's "killing the broadcast forces an optimistic local write
(banned)" aside.** That ban was scoped to KILLING the broadcast; here the
broadcast stays (server-owned mirror + cross-device + only backward
mover) and optimism just removes the originating device's own latency.
Corrected the wrong "(banned)" sub-claim at its source in the
freeze-contract DESIGN_NOTES entry. Tradeoff: failed POST leaves local
ahead, not reverted (a revert would clobber a concurrent forward
advance); bounded to already-read rows, re-aligns on next forward write /
`/me` / join-reply.

### 2. Multiline compose → one PRIVMSG per line (`305dbcb` fix, `ddcbab4` docs; bundle `DAowd1rL`)
vjt report: a multiline message (Shift+Enter / pasted block) bounced with
an "invalid" error and nothing sent. Cause: the whole body went as one
PRIVMSG; the server correctly rejects an embedded CR/LF as `:invalid_line`
(frame delimiter).

Fix: split client-side — new pure util `messageLines.ts`
`splitMessageLines` (splits on CRLF/CR/LF — all forbidden on the wire so
all must split; drops blank lines) + shared `sendBodyLines` in
`compose.ts` over the three free-text sites (privmsg, /me one-ACTION-per-
line, /msg). Division of labor: CLIENT owns newline splitting, SERVER
keeps owning 512-byte length splitting (`line_split.ex`). Single-line
body unchanged. Accepted edges: non-transactional partial send on
mid-fan-out failure (full draft preserved → retry re-sends); empty /me
now a no-op.

### 3. Send-relatch: hide the unread marker on a focused send (`ccbae53` fix, `94530b3` docs; bundle `CEG1msUy`)
vjt report (2026-06-09): a `── 1 unread ──` divider that didn't disappear
when he sent a new message in the focused channel. NOT a regression — it
is the cp56 freeze contract working as written: the in-pane divider reads
the FROZEN `markerCursorId`, which re-latched only on a focus acquisition
(channel-switch / visibility-return), and a send is neither.

The conflict is genuine and both halves are vjt's: "don't move the
divider while I read" (cp56) vs "hide it when I send" (now). A send and a
passive scroll-settle BOTH advance the live cursor through the same
`setReadCursor` → indistinguishable at the cursor, so the send must mark
itself. `sendMessage` publishes its channel-key on a new `lastOwnSend`
EVENT signal (`equals: false` — same-channel repeats must re-notify);
`ScrollbackPane` watches it and runs the identical re-latch the
channel-switch path uses, keyed to THIS pane. Fired ONLY from the
own-send path, so passive advances (scroll-settle, cross-device) stay
frozen — the freeze holds for everything but the operator's own send.

vjt pushback mid-build ("do we really need this lastOwnSend? looks
awful") → trimmed from a `{key,id}` object to a bare key string (the
effect re-latches to the live cursor, never the send id). One signal, one
writer, one reader. The `equals: false` dedup gap was caught in self-
review (the parallel review agent had called it benign — it wasn't: a
send → switch-away → peer-msg → switch-back → reply sequence re-shows
then fails to re-hide).
