## What shipped

The "server-routed numeric replies are durable scrollback rows in
their natural window" architecture, plus the mIRC text-formatting
parser, plus the `$server` window slash-only compose box, plus a
priority-chain rewrite of EventRouter NOTICE handling that gives
ChanServ inline + per-user NOTICEs the natural window they should
have always landed on.

### Goal

Pre-CP13 cicchetto stored server numerics in an in-memory ephemeral
signal store (`numericInline`) and rendered them in a separate bottom
pane below the scrollback. Three problems:

  - Not durable: a numeric you weren't connected for never appeared.
  - Not replayable: REST scrollback didn't include them.
  - Not in the routed window's flow: a 404 on `#sniffo` lived in a
    parallel ephemeral lane instead of inline with `#sniffo`'s history.

### Architecture

Server side:

  - `Scrollback.Meta.@known_keys` extended with `:numeric` + `:severity`
    (no migration ā€” `meta` is a `:map` JSON column).
  - `Grappa.Session.NumericRouter` rewritten: drops the per-numeric
    lookup tables (`@channel_param_numerics` / `@nick_param_numerics`)
    + the `last_command_window` `:active` resolution. New flow: label
    override ā†’ delegated set ā†’ `@active_numerics` deny list ā†’
    `{:server, nil}` ā†’ param scan (channel-prefix ā†’ nick-shape ā†’
    fallback `{:server, nil}`).
  - `Grappa.Session.EventRouter` NOTICE-non-channel-target replaces
    the single greedy "ā†’ $server" with priority chain:
    1. ChanServ + body matches `[ #chan ]: msg` ā†’ strip prefix +
       persist on captured channel.
    2. `*Serv$` sender ā†’ `$server`.
    3. Sender hostname (contains `.`) ā†’ `$server`.
    4. Regular user nick ā†’ persist on `channel = sender_nick`
       (query window).
    5. Anonymous sender / unattributable ā†’ fallback `$server`.
  - `Grappa.Session.Server` numeric handler: replace the
    `numeric_routed` PubSub broadcast with a `:persist` effect
    producing `:notice` rows carrying `meta = %{numeric, severity}`.
    Routed via the regular `apply_effects` pipeline ā†’ broadcasts on
    the per-channel topic exactly like any other scrollback writer.

Cicchetto side:

  - DELETE `lib/numericInline.ts` + tests + the `.numeric-inline-pane`
    JSX + the `numeric_routed` branch in `lib/userTopic.ts` (-511 lines).
  - ScrollbackPane: `:notice` rows with `meta.severity === "error"`
    get `.scrollback-notice-error` (red via `--mode-op`).
  - Sidebar + BottomBar: `$server` window mirrors channel-pattern's
    3 unread badges (msg / events / mention).
  - Shell: drop the `kind !== "server"` ComposeBox guard. compose.ts:
    `$server + privmsg-shape draft` rejects with friendly error
    "Server window accepts only slash-commands. Try /raw <line>".
  - NEW `lib/mircFormat.ts`: tokenizer for the 6 toggle codes
    (`\x02`/`\x1d`/`\x1f`/`\x16`/`\x0f`) + the color code
    (`\x03[fg[,bg]]`) + the 16-color mIRC palette. Output `Run[]`,
    rendered by `MircBody` as one `<span>` per Run with toggle
    classes + inline-style fg/bg.
  - PRIVMSG / NOTICE / ACTION bodies render through MircBody.

### Caveat S5 ā€” closed

The plan caveat: `{:query, nick}` numeric routes to `channel = nick`,
but cic only joins the per-(slug, nick) WS topic when the query
window is open. First-contact numerics could vanish live (DB-only).

Validated in S11 e2e (`/msg <ghost> hi` ā†’ 401 ERR_NOSUCHNICK). The
loop closes naturally: cic's `/msg` handler opens the query window
client-side BEFORE the server round-trip completes, so the
per-(slug, nick) WS subscription is live by the time the routed
:notice broadcast fires. The row arrives in the open query window
scrollback as a `.scrollback-notice-error`. Spec passes deterministic.

No server-side `query_window_opened` broadcast needed.

## Lessons

  - **Orchestrator override on plan**: orchestrator initially proposed
    "new `:numeric` schema kind + migration" which contradicted the
    CP13 plan's explicit "REUSE `:notice`". Re-anchored to the plan
    immediately on operator correction. Plan is the authority, even
    against orchestrator directives.
  - **State leak in full e2e suite**: S10 passed standalone but
    failed in full-suite ordering because `selectChannel(awaitWsReady=true)`
    waited for a fresh JOIN-self line that never came (the bootstrap
    one had already been consumed by earlier tests). Fix: pass
    `awaitWsReady: false` when the channel is known to be already
    joined and just refocusing.
  - **`mix deps.audit` is part of `scripts/check.sh`**: bandit 1.10.4
    had been in `mix.lock` since the cloak/auth deps landed (commit
    `16b0b61`), but only surfaced when this cluster's gate sweep
    forced a clean check.sh. Bumped 1.10.4 ā†’ 1.11.0 (4 CVEs cleared).
    Reinforces feedback_landed_claim_evidence: every gate matters
    every time, not just when "we touched that area."
