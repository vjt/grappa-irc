### Discovered bugs (2 fixed live, 3 outstanding)

**FIXED via reload:**
1. CRLF footgun — `send_line/2` callers had to remember `\r\n`; my
   CTCP reply forgot it; second outbound frame concatenated +
   garbled the wire. Fixed at the transport boundary in
   `Grappa.IRC.Client.handle_call({:send, line}, ...)` via
   `ensure_crlf/1`. All callers now safe-by-default.
2. CTCP query invisible in cic — server now emits a `:notice`
   persist effect with body "CTCP VERSION query → grappa <vsn>"
   so cic can show inbound CTCP traffic (subject to Outstanding
   Bug #2 below).

**OUTSTANDING (uncommitted in tree, see `git status`):**

1. **`Grappa.version()` returns 0.0.1 not 0.1.0.** `mix.exs`
   `@version "0.1.0"` only takes effect on full `mix compile`
   (regenerates `.app` resource file). `Phoenix.CodeReloader.reload/1`
   recompiles `lib/*.ex` but NOT mix.exs → `Application.spec(:grappa,
   :vsn)` reads stale 0.0.1 from the loaded `.app`. **Next
   `scripts/deploy.sh` (cold deploy) will fix this.** No code
   change needed; cluster's hot-reload story has a documented
   limitation (struct-shape changes, mix.exs changes, supervision-
   tree changes need cold deploy — already in plan B9 image-label
   rules).

2. **Inbound CTCP NOTICE didn't auto-open cic DM window.**
   Investigation found:
   - The dm-listener arm in `cicchetto/src/lib/subscribe.ts` only
     auto-opens for `kind: "privmsg" | "action"` — NOTICE was
     dropped (legacy "Bug A fix" assumed all notices on own-nick
     topic were service notices like NickServ).
   - vjt's correct guidance: "do for notice the same fucking thing
     we do for privmsg" — peer NOTICEs on own-nick topic ARE
     legitimate DMs and should auto-open.

   **In-flight uncommitted edits (4 files, NOT TESTED YET):**
   - `lib/grappa/scrollback/message.ex` — added `:notice` to
     `@dm_with_eligible_kinds` (was `[:privmsg, :action]`). Allows
     notice rows to carry `dm_with` per the schema changeset.
   - `lib/grappa/scrollback.ex` — extended `dm_peer/4` guard to
     include `:notice`. Same DM-detection rule applied to peer
     notices targeting our nick.
   - `cicchetto/src/lib/subscribe.ts` — added a `kind === "notice"
     && sender !== ownNick` arm in `installDmListenerHandler`. Peer
     notices auto-open sender's window same as PRIVMSG; own
     outbound NOTICE echoes drop silently.
   - `cicchetto/src/__tests__/subscribe.test.ts` — replaced the
     old "NOTICE dropped" test with two new tests: (a) inbound
     peer NOTICE auto-opens, (b) own outbound NOTICE drops.

3. **Manual cic refresh required to see persisted notice rows.**
   Existed before Outstanding #2 fix; may or may not be
   independently fixed once #2 lands. Symptom: vjt sent CTCP
   VERSION queries; rows DID land in DB; cic only showed them
   after page refresh. Likely the same root cause as #2 (no
   auto-open path → no subscription → broadcast missed).
