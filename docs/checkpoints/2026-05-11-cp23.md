## Bucket plan (vjt 2026-05-11)

vjt's stated next ask: **hot code reload for grappa dev workflow**
("I FUCKING WANT TO START USING CODE RELOAD"). Currently every `.ex`
edit requires `scripts/deploy.sh` (~20-30s release rebuild + lost
ETS state + lost session connections). Want Phoenix's standard dev
workflow: edit → save → seconds later live in BEAM, sessions
preserved.

Brainstorm-first per CLAUDE.md Foundation rules. The 5 design
questions queued by the orchestrator transcript at
`/tmp/orchestrate-next.txt` are the first turn:

1. **Dev image vs prod image** — Phoenix dev needs `mix` in container,
   current Dockerfile is multi-stage release that strips it. Options:
   (a) Dockerfile.dev sibling, (b) single Dockerfile with two
   targets, (c) dev runs upstream `elixir:1.19` directly.
2. **Code reload mechanism** — Phoenix `code_reloader: true` covers
   request-path reload (controllers/views). For non-request paths
   (IRC parser, EventRouter, Session.Server) need either `mix
   phx.server` boot with `:fs` watcher, or `recompile` from IEx, or
   module-level `Code.compile_file/1` (brittle).
3. **Process state survival** — code reload doesn't restart processes;
   most changes safe but struct field additions need `code_change/3`
   migration. Policy?
4. **Bind mount strategy** — `lib/`, `test/`, `config/` need to
   bind-mount from host so editor edits trigger in-container `:fs`
   watcher. compose.dev.yaml profile?
5. **Cic side already hot** — Vite dev has HMR via
   `scripts/bun.sh run dev`. Confirm scope is Elixir-only.

Suggested cluster shape (still needs vjt sign-off):

- Worktree `cluster/code-reload`
- B1 brainstorm + design pin (1 file in `docs/plans/`)
- B2 dev container target (Dockerfile or multi-target)
- B3 dev compose service running `mix phx.server` with bind-mount
- B4 `scripts/dev.sh` to bring up dev stack
- B5 docs in CLAUDE.md "How to run scripts" dev-vs-prod distinction
- B6 smoke verify: edit a controller → response change without
  container restart
