# CP15 — event-driven windows

Cluster opened 2026-05-07 immediately after CP14 closed at `620ef05`.
Implements the architectural shift pinned in
[`docs/plans/2026-05-07-event-driven-windows.md`](../plans/2026-05-07-event-driven-windows.md):
**one state model, on the server. cicchetto NEVER assumes state
transitions; it consumes typed JSON events.**

The premise: cic's optimistic mutation pattern (sidebar entries appear
on POST /channels success, members pane fetches once via REST and
caches) hides server reality. When IRC says invite-only, when upstream
goes parked, when the WS reconnect race drops the post-deploy
`members_seeded` broadcast, the UI silently lies — empty members,
ghost windows, scrollback rendered for channels we can't post in.

Window LIFECYCLE moves to event-driven (server emits `:joined`,
`:join_failed`, `:parted`, `:kicked`, `:window_state`). Window
CONTENT moves to event-driven (members + topic + modes already there;
cic's `loadedChannels` REST gate goes away). Window VISIBILITY stays
user-action-driven (PART/× moves to archive; KICK/T32 stays active in
greyed state).

Detailed bucket breakdown + TDD steps + decisions in
`docs/plans/2026-05-07-event-driven-windows-impl.md` (sibling, opens
with this checkpoint).
