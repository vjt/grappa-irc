### C3 — pin SQLite `synchronous` + `foreign_keys` PRAGMAs

`config/runtime.exs:31-55`, `config/dev.exs:3-10`,
`config/test.exs:3-26`.

Today's `exqlite` default of `synchronous: :normal` is the correct
choice for WAL and `foreign_keys: :on` is the verified default —
but **right by accident**. A dep major-version flip that disables
either default silently turns every prod commit into a fsync-deferred
best-effort write OR breaks the CASCADE chain (visitor reap walks 8
tables). Pinned both in all three Repo configs.

Zero runtime behavior change today; insurance against future dep
upgrades.

### H6 — closed-set fallthrough in Networks.{connect,disconnect,mark_failed}

`lib/grappa/networks.ex`.

Each function pattern-matched a subset of
`Credential.connection_states/0` without explicit fallthrough. If
the enum gains a 4th state (e.g. SASL-gated `:locked`), the typespec
would lie to Dialyzer (clauses exhaustive on CURRENT enum but not
FUTURE) and runtime would silently `FunctionClauseError` rather than
the typed `{:ok|:error, _}` contract.

Per `feedback_no_silent_drops_closed`: explicit fallthrough that
RAISES `ArgumentError` carrying `inspect(other)` for actionable
diagnostics. Mirrors `Scrollback.subject_where/2` B5.4 L-pers-2
precedent.

Three regression-guard tests in `networks_test.exs` describe block
`REV-B H6 — closed-set fallthrough on unknown connection_state`
(fabricated `:locked` atom on a struct literal — no DB write since
Ecto.Enum cast at the schema rejects pre-write, distinct from
function-clause discipline).

### H17 — Identifier.canonical_channel/1 consistency in delete paths

`lib/grappa/scrollback.ex` (`delete_for_channel/3` + `delete_for_dm/3`)
+ `lib/grappa_web/controllers/archive_controller.ex` (dispatcher).

Write side canonicalised channels via `Identifier.canonical_channel/1`
(sigil-aware). Delete side raw-downcased. ASCII channels agree
today, but any future canonicalisation extension (Unicode casefold,
leading-`!` strip) would silently make the delete miss its target
rows.

Single-sourced via `Identifier.canonical_channel/1` at three sites:
- `delete_for_channel/3` canonicalises + uses plain `==` against
  the stored canonical column (no `lower()` fragment needed).
- `delete_for_dm/3` routes through `canonical_channel/1` (no-op on
  nicks; boundary consistency). The `lower()` SQL fragment stays —
  `dm_with` is the case-preserved NICK column per
  `message.ex:252-254` design, and `lower()` is the correct
  case-insensitive nick compare.
- `ArchiveController.delete/2` canonicalises the URL `target` BEFORE
  sigil dispatch — boundary-level so both branches observe same
  normalised input.

StreamData property test in `scrollback_test.exs` pins write/delete
parity across mixed-case channel-shape names.

### H18 — covering index for Scrollback.list_archive/3 + migration

NEW `priv/repo/migrations/20260522073826_add_archive_covering_indexes.exs`.

`list_archive/3` does `GROUP BY COALESCE(dm_with, channel)` with no
covering index — full per-subject per-network row scan + temp B-tree
sort at every archive open. N×log(N) per open on a heavy user with
50k+ messages. Costs nothing today (small row counts) but the bound
is wrong.

Two expression indexes:
- `messages_archive_user_idx`:
  `(user_id, network_id, COALESCE(dm_with, channel), server_time)`,
  partial `WHERE user_id IS NOT NULL`.
- `messages_archive_visitor_idx`: mirror for visitor subject.

Trailing `server_time` makes both covering for BOTH archive
`MAX(server_time)` aggregate AND DM-fetch `ORDER BY server_time DESC`
shape — the planner sometimes picks the new index over the older CP14
H1 `(subject, network_id, dm_with, server_time)` composite for the
OR-arm DM fetch. Both choices preserve the H1 "no cross-subject scan"
invariant; the existing CP14-H1 regression test was loosened to
accept either subject-leading composite while keeping the `refute`
against the subject-less index (CLAUDE.md "never assert buggy
behavior" satisfied: the original invariant was "subject-LEADING",
specific index name was always implementation detail).

EXPLAIN QUERY PLAN regression test pins planner selection on the
archive query shape.

Pre-fix the call lived in `handle_info({:EXIT, client_pid, _})` +
`do_start_client/2`'s inline-start-failure branch only. Any
non-Client-EXIT crash (server-internal callback raise, mailbox

moduledoc CLAIMED `record_failure` fired on "any non-`:normal` exit"

Funneled the call into `Session.Server.terminate/2`'s new
abnormal-reason clause. `terminate/2` now has three clauses:
`:shutdown | {:shutdown, _}` (graceful — no bump), `:normal`
(operator-driven — no bump), and the catchall (every other crash
class → single bump). Removed both pre-fix call sites; verified
single-bump invariant under all six abnormal stop reasons

from a hypothetical non-Client linked proc). `:brutal_kill` +

### H13 — Visitor.touch_changeset monotonicity + expire_changeset split

### H14 — Visitors.commit_password + update_nick race protection
