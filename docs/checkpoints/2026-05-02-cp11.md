## S20 — 2026-05-03 — Hotfix: Session.Backoff exponential reconnect

Mid-session emergency interruption. Operator (vjt) flagged that
azzurra had K-lined the bouncer's IP — three smoke-test visitors
(smoketest1, smoketest3, smokebrowser1) had been hammering azzurra
since the deploys this evening, ~6 attempts/min combined for hours.

Root cause: `Grappa.IRC.Client`'s `@connect_failure_sleep_ms` throttle
(S18 H1 fix) only covered `handle_continue/2`'s connect-fail branch.
A K-line behaves as TCP accept then immediate close — the connection
"succeeds" past handle_continue, then `:tcp_closed` arrives →
`{:stop, :tcp_closed, _}` immediately, NO throttle. Each crash triggers
a `:transient` respawn of `Session.Server`, the new Session spawns a
fresh Client, the new Client connects, the upstream closes again, the
loop repeats at restart-rate. azzurra k-lined as a defense.

### Stop the bleeding

`docker compose -f compose.prod.yaml exec -T grappa bin/grappa rpc
'IO.inspect Grappa.Visitors.reap_by_network_slug("azzurra")'` →
`{:ok, 3}`. CASCADE wiped sessions + visitor_channels + messages for
the three rows. Restarted the prod container so Bootstrap saw an
empty `Visitors.list_active/0` and didn't respawn anything. Hammering
loop dead.

### Root-cause fix shipped (`020cd60`)

New `Grappa.Session.Backoff` ETS-backed module owned by the
Application supervisor (named table `:session_backoff_state`). Per-
`(subject, network_id)` failure counter SURVIVES `Session.Server`'s
`:transient` restart cycle — the new init/1 reads it before spawning
the Client and defers via `Process.send_after` if non-zero. Reads
direct ETS lookup; writes funnel through a cast for read-modify-write
consistency.

Wiring in `Session.Server`:

  * `init/1` sets `Process.flag(:trap_exit, true)` so a Client crash
    arrives as a message instead of killing this process via the
    link — gives us a chance to record the failure before stopping.
  * `handle_continue({:start_client, _}, _)` reads `Backoff.wait_ms/2`,
    defers spawn via `send_after` if non-zero, runs immediately if
    zero. Logs `delay_ms` + `failure_count` for observability.
  * `handle_info({:start_client_after_backoff, _}, _)` does the
    deferred spawn.
  * `handle_info({:EXIT, client_pid, reason}, _)` when from our
    Client → `record_failure` + `{:stop, {:client_exit, reason}, _}`
    (wrapped reason distinguishes "Client died on me" from
    "I asked Client to die").
  * `handle_info({:EXIT, _, :shutdown|:normal}, _)` → propagate, no
    failure record (operator-issued).
  * 001 RPL_WELCOME handler calls `Backoff.record_success/2` —
    upstream accepted us, prior failures are stale, ladder resets to
    count=0.
  * `do_start_client/2` failure path also records (Client.start_link
    returning `{:error, _}` is the same kind of failure for backoff
    purposes).

### Curve at defaults (config/config.exs)

`base_ms: 5_000, cap_ms: 30 * 60 * 1_000` — `base × 2^(count-1)`
capped, with ±25% jitter to avoid herd-respawn alignment.

| count | nominal wait |
|-------|--------------|
| 1     | 5s           |
| 2     | 10s          |
| 3     | 20s          |
| 4     | 40s          |
| 5     | 80s          |
| 6     | 160s         |
| 7     | 320s         |
| 8     | 640s         |
| 9     | 1280s        |
| 10+   | 1800s (cap)  |

After ~5 fails we're already past 1m wait; after 10 we're at the
30-min cap. A k-line bouncer would idle for the cap duration between
attempts instead of hammering — the IP K-line expires (typically 24h
on azzurra) without further escalation. Test config shrinks to
`base_ms: 5, cap_ms: 100` so test runtime doesn't drag.
