## Session 2026-06-23/24 — keyboard-free nick completion → swipe gestures ✅ SHIPPED

A multi-iteration cic UX arc. Goal: make nick tab-completion usable on a
stock mobile keyboard (no Tab key), so the custom IRC keyboard becomes
optional. Brainstorm → plan → subagent-driven execution → dogfood-driven
trigger pivots. Commits `314f5a4..2f4ae03`, all FF-merged + deployed hot.

**1. irssi-exact nick completion + latent-bug fix** (`7f219e0`, `af76dbb`).
Rewrote `tabComplete` (`cicchetto/src/lib/compose.ts`): positional suffix
(`": "` at line start / `" "` mid-sentence), cycle space
`[match0..matchN-1, <typed>]` so the last tab reverts to the typed text
before wrapping, and **range-based** continuation (anchor `[start,end]` +
`lastInsertion` match) — replaces word-equality, which broke the instant a
suffix sat after the caret, and lets a re-tap inside the inserted nick
count as the same cycle. **Latent bug found while scoping:** in-app cycling
NEVER worked — `setDraft` nulls the cycle anchor (correct: a real edit
breaks the cycle) and both callers (`Shell.tsx`, `KeyboardHost.tsx`) called
`setDraft(result.newInput)` right after `tabComplete`, nulling it every
time; the old unit tests "passed" only by bypassing `setDraft`. Fix:
`tabComplete` writes the draft itself via `writeState` (doesn't null the
cycle); callers drop their `setDraft` and only place the caret.
discard-on-keystroke needs no new code (`onInput → setDraft` already nulls).

**2. Trigger: double-tap → swipe-right** (`fb0d428`, `0dfcf64`, `200962c`).
Shipped double-tap first; dogfood confirmed it collides with the OS
word-select (the collision flagged theoretical at design time was real).
iOS double-tap-select is a system gesture recognizer that can't be reliably
`preventDefault`'d, so pivoted to **swipe-right**. **Code review caught a
[Critical]:** Solid *delegates* `touchstart/move/end` to a single PASSIVE
listener on `document`, so a JSX `onTouchMove` `preventDefault()` silently
no-ops (jsdom doesn't enforce passive → suite stayed green). Fix: bind the
touch listeners on the textarea via a `ref` + `addEventListener`, touchmove
explicitly `{passive:false}`, `onCleanup` to remove. Pure `lib/swipe.ts`
reducers, unit-tested; gesture dogfood-only.

**3. Regression + fix: shell-drag** (`d1cb25a`). First swipe dogfood:
dragging from the input dragged the whole shell. Root cause — the compose
textarea was the ONE touchable control with no explicit `touch-action`
(defaulted to `auto`); the non-passive touchmove routed the gesture
main-thread and iOS drove its chrome/overscroll from the unguarded `auto`.
Per UX-3 UNDEC R3 (`.shell-mobile { touch-action: none }` blocks
chrome-reveal; scroll regions re-assert their axis), chrome that isn't a
scroll container must be `none`. Fix: `touch-action: none` on
`.compose-box textarea`. Confirmed by vjt on device.

**4. Swipe up/down → history recall** (`2f4ae03`). Extended to all three
keyless affordances: swipe RIGHT = Tab (`tabComplete`), swipe UP = ArrowUp
(`recallPrev`), swipe DOWN = ArrowDown (`recallNext`). Unified the reducers
into one direction classifier — `swipeDirection` (dominant axis,
diagonal→null) + `dragAxis` (mid-drag axis claim) — replacing the
right-only booleans. Handler locks to one axis on the first move past slop,
so a gesture is horizontal-complete OR vertical-recall, never both. `left`
classified but unmapped (reserved for back-cycle). No browser conflict
because of the `touch-action: none` fix. Confirmed by vjt on device.

## Session 2026-06-25/26 — grappa-irc#81 synthetic-window 404 ban-cascade ✅ SHIPPED

Dispatched bug (filed by vjt-claude from the IRC session). The cic
selection effect fired `loadInitialScrollback` for EVERY focused window,
so the synthetic `$home` status buffer emitted
`GET /networks/$home/channels/$home/messages` → 404. On m42 the repeated
404 trips fail2ban `http-404` → pf block → `recidive`, banning a real
operator (`emme\k`) at the network layer from one IP. A server-side
`ignoreregex` stopgap was already live; this is the client root-cause fix.

**Fix** (`4a6577a`, docs `140a9bf`). Not just `$home`: same class 404s for
`$admin` and `mentions` (empty channelName → `//messages`). Gated on a
POSITIVE predicate `kindHasScrollback(kind)` (true only for
channel/query/server) in `cicchetto/src/lib/windowKinds.ts` — an
exhaustive `Record<WindowKind, boolean>` so a new kind fails to compile
until classified. Rejected the issue's "skip any `$`-prefixed window"
hint: `$server` IS scrollback-backed (NumericRouter rows), that heuristic
would have suppressed real server-pane history. The predicate replaced the
`channel||query||server` literal duplicated 3×: the two ScrollbackPane
mount guards (`Shell.tsx`) and the `saveLastFocused` restore gate
(`selection.ts`, same effect body — restorable set ≡ scrollback-backed set,
both = "has a real (network, channel) identity").
