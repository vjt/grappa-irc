# CP29 — server-side read-state cluster (invariant flip)

Opened 2026-05-13. Brief at
`docs/plans/2026-05-13-server-side-read-state.md`. Cluster mandate
(vjt 2026-05-13): per-bucket discipline at the BRANCH level; cold
cutover held to R-Z so production sees the cluster as one atom.

## What flipped

Pre-cluster CLAUDE.md "key invariants" said:
*"No server-side `MARKREAD` / read cursors. Read position is
client-side only."*

Post-cluster:
*"Read state is server-owned, per (subject, network, channel).
Cursor stored as `last_read_message_id` (FK to `messages.id`)."*

cic's localStorage `read_cursor` / `rc:`-prefixed keys are gone; cic
reads cursor from the `/me` envelope on cold load + per-channel from
the WS topic join reply + cross-device via `read_cursor_set` typed
events. cic POSTs the operator's current position via
`Grappa.ReadCursor.set/4` (last-write-wins) on every settle event
(focus-leave + browser-blur in selection.ts).

## Decisions deferred (with wiring sites named)

- **Auto-set cursor on operator's own POST**: not wired. The
  selection.ts focus-leave model (with sibling browser-blur arm) is
  the canonical "I've moved on" signal and is uniform across own-msg
  vs peer-msg vs scroll-up scenarios. Wiring site if needed:
  `Grappa.Scrollback.persist_event/1` (or controller layer) — wire
  after the persist, before the broadcast, so the broadcast carries
  the new cursor.
- **Explicit mark-as-unread UI**: server verb (`Grappa.ReadCursor.set/4`,
  last-write-wins) is the wire surface; cic exposure (slash command,
  right-click) is a follow-up cluster.
- **Scroll-settle cursor derivation**: cic currently sets cursor to
  scrollback tail on settle. Reading actual visible row from scroll
  position and setting the cursor there is a follow-up.
- **Mention click cursor-rewind UX**: requires extending
  `MentionsBundle` wire shape with message ids + a one-shot
  scroll-to verb in `ScrollbackPane`. Separate cluster.
