## Pending at checkpoint open

### Phase 3 — cicchetto PWA bootstrap (in-monorepo)

CP09 S2.5 (2026-04-26) corrected a misreading from CP08 + the
opening of CP09: cicchetto is **NOT** a separate repository.
README line 1 of "What" reads "Two components, one monorepo." The
"two facades, one store" DESIGN_NOTES decision is about REST+WS
vs IRCv3-listener facades over a single scrollback DB — orthogonal
to the repo-organization question, which the README answered as
monorepo from the start.

Phase 3 lives at `cicchetto/` as a sibling of `lib/` + `test/` in
this repo.

Per Roadmap:
- [ ] PWA shell, manifest, service worker (install-eligible day 1)
- [ ] Login flow → token → connect `/socket/websocket`
- [ ] Channel list + scrollback fetch on select
- [ ] Send message

**Client-stack decision (LANDED 2026-04-26 in conversation, to be
written into DESIGN_NOTES at Phase 3 start):**

| Slot | Pick | Why |
|------|------|-----|
| Framework | **SolidJS** | Fine-grained reactivity → high-frequency WS updates (every PRIVMSG = state update) only re-render the affected DOM nodes, no virtual-DOM diff, no component re-runs. ~7kb runtime. |
| Language | **TypeScript** | Wire-shape contract on JS side — mirrors `@spec` + Dialyzer discipline server-side. SolidJS is TS-native. |
| Build / dev | **Vite** | Universal default; Solid plugin first-class. |
| Tooling runtime | **Bun** | Single binary (install + dev + TS + JSX + tests + bundler), ~10× faster than pnpm/npm. Build-only — nginx serves static `dist/` in prod, so runtime risk surface = zero. |
| Router | **`@solidjs/router`** | First-party, file-based routes, lazy loading. |
| Linter / formatter | **Biome** | Single fast tool replacing ESLint + Prettier. |
| Tests | **`@solidjs/testing-library` + Bun's built-in test runner** | Component + integration; deferred until first non-trivial component. |
| Theming | **CSS variables + theme files in `cicchetto/src/themes/`** | Day-1 barebones (system mono, fixed line height, no chrome). Future: irssi / mIRC / xchat theme packs. Theming infrastructure is designed-in from the start; only one theme exists at walking-skeleton landing. |

**Auth model:** `POST /auth/login` → `{token}`, store in
localStorage, attach as WS `connect` query param. Phase 2 already
has `Grappa.Accounts.Session` (DB-backed token sessions) — Phase 3
is wiring the existing surface, no server-side auth changes.

**Deploy shape (single hostname, no CORS):** new DNS record
`grappa.bad.ass A 192.168.53.11` via Technitium API
(token at `/srv/dns/.env`). nginx terminates HTTPS, serves
`cicchetto/dist/` static, proxies `/api` + `/socket` to the
grappa container. Two new compose services:

- `cicchetto-build` — slim Bun image, one-shot, runs `bun run build`
  into a named volume on container start, exits.
- `nginx` — mounts the named volume read-only; proxies `/api` +
  `/socket` to grappa; serves SPA fallback (`index.html`) for
  client-side routes.

**Cross-stack coupling:** wire-shape contracts grappa ships are
the cicchetto contract. Phase 3 will surface gaps in grappa's
REST surface (notably **missing `GET /networks` + `GET /networks/:nid/channels`
listing endpoints** — the channel-list view needs them). Land
those on `main` directly when small; bigger ones in a
`phase3-server-gaps` worktree.
