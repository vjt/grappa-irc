### Unread-divider freeze (`a847b59` impl+unit, `5a8a6a1` docs, `005fe68` e2e)
vjt report: scrolling an unread block yanked the in-pane `── N unread ──`
divider down under your eyes — the `rows()` memo read the LIVE
`getReadCursor`, so a scroll-settle advance (or a cross-device
`read_cursor_set`) re-ran it mid-read and shrank/removed the marker.

Fix (`cicchetto/src/ScrollbackPane.tsx`): the divider now derives from a
FROZEN snapshot signal `markerCursorId` (frozen BOTTOM boundary, sibling
to the pre-existing frozen `sessionTopId` TOP boundary). It re-latches to
the live cursor ONLY on a focus acquisition — channel-switch (key effect)
+ tab/app visibility-return (**option b**, vjt's pick over "channel-switch
only"). A cold-latch effect mirrors the `sessionTopId` cold-mount latch to
pick up the cursor when it hydrates after mount; its `markerCursorId() !==
null` guard is read FIRST so a post-latch live advance can't re-run it
(freeze-safe under Solid's per-run dependency model). The live signal map
stays the single source of truth for sidebar badges + `selection.ts`
unread counts — only the in-pane divider freezes.

**UNIFORM freeze, accepted tradeoff (vjt: "consistency"):** cic cannot
distinguish an own scroll-settle echo from a peer's cross-device
`read_cursor_set` at the applier boundary (same wire bytes, no client_id
tag), so the freeze is uniform — send-in-window AND cross-device reads
also defer their divider collapse to the next refocus, not just scroll.
Distinguishing them would need client_id tagging on the broadcast
(server + wire change); rejected as heavier than the problem. Rejected
also "just suppress the broadcast": the server echo is what keeps cic
mirroring server-owned state; killing it forces an optimistic local write
(banned) and breaks re-focus advance.

Revises the CP29 R-4 "Bug A" contract (marker disappeared immediately on
any live advance). Full decision + alternatives: **DESIGN_NOTES
