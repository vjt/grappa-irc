## What shipped

### vitest location-teardown flake — fixed (`bcb813b` + `c7401af`)

Root cause was NOT the blamed `windowState.test.ts` (victim, not
offender): **jsdom 29 ships a real ws-backed WebSocket** that opens
actual TCP connections. Any test file where `lib/socket`'s module-level
token effect fires with a non-null bearer (phoenix unmocked) attempts a
real connect; the failed connect arms phoenix's reconnect-backoff timers
on the Node event loop, and a timer outliving that file's jsdom env
dereferences the torn-down `location` global — crashing whichever test
file runs next in the worker.

Fix at the boundary (no per-file whack-a-mole): `setupTests.ts`
re-installs an **inert WebSocket stub** before every test (same pattern
as the localStorage stub, survives `vi.unstubAllGlobals()` in file
afterEach hooks). Construction succeeds, nothing connects, no event
fires, phoenix never arms a reconnect — the leak class is gone and unit
tests stop making real network attempts. Guard test
`inert-websocket.test.ts` (TDD RED→GREEN: real impl fires error/close +
throws InvalidStateError on early send; stub stays inert).

Review verdict **SOUND** (full phoenix 1.8.5 transport-contract audit;
LongPoll-fallback escape refuted — `longPollFallbackMs` unset and the
stub wins transport selection; teardown fast-path verified — `close()`
→ readyState 3 means zero retry timers). Review LOW fixed in follow-up
commit: `serverSettings.test.ts` called `vi.unstubAllGlobals()` in a
describe-level **beforeEach** (runs after setup's, resurrecting the real
WebSocket for that block) — moved to afterEach.
