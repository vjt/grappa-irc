### S1.1 — MembersPane prefix clip (commit `2ffc29f`)

Symptom: members panel showed only `@` / `+` characters, full nicks
invisible. Root cause: Spec #5 click-button (`width: 100%` block-level)
+ pre-existing `::before { content: "@" | "+" }` inline prefix → button
wrapped to a new line below prefix → li `overflow: hidden` clipped the
wrapped button. Plain rows masked it (prefix is just space).

Fix: render prefix as first character of button's text content via
new `lib/memberSigil.ts` (lifted from `ScrollbackPane.tsx`'s former
module-private helper). Drop CSS `::before` rules. Inline flow,
no wrap, no clip. ScrollbackPane reuses the same helper (one source
of truth per CLAUDE.md "implement once, reuse everywhere").

Tests: vitest extended (`toContain("@vjt")` style — assertion now
catches the regression class because prefix lives in DOM textContent
not CSS). New Playwright e2e `members-prefix-regression.spec.ts`
asserts `.member-op .member-name` text equals `@<nick>` AND
boundingBox width > 30px (catches the wrap+clip; jsdom is blind to
`::before` and to layout). 713 vitest pass, 1411 mix tests pass,
scripts/check.sh EXIT=0.

### S1.2 — Focused-window marker + switch-target scroll (commit `dc3042b`)

Symptom A (marker): operator types `/msg <bot> body`, bot replies a
beat later in the focused query window — yet "1 unread message" marker
spawns between own-msg and reply.

Symptom B (scroll): switch from a DM (where own-msg + peer-reply just
landed) to a TALL channel — viewport stuck at top instead of bottom.
Second visit works.

Root cause (BOTH bugs): `markerRef` retained the disposed DOM pointer
across window key changes. Solid's let-bound refs assigned via JSX
`<div ref={markerRef}>` inside `<For>`-rendered subtrees DON'T auto-
null on unmount — the variable retains the stale node. Key createEffect
took the marker branch on the wrong window, called `scrollIntoView`
on the disposed node (no-op), never fell through to scrollTop=
scrollHeight. Marker bug shared the leak: focused-window arrivals had
`server_time > cursor` (cursor only advances on own-msg) so the rows
memo's filter included peer reply → marker injected.

Fix (3 parts, ScrollbackPane.tsx only):
1. `markerRef = undefined` synchronously at key change in createEffect
   — closes the leak before scroll branch evaluates.
2. `queueMicrotask` defer of the scroll decision — Solid commits the
   new window's rows BEFORE we read markerRef. New marker (if any)
   reassigns ref via `<div ref={markerRef}>`; otherwise scroll-to-bottom.
3. New `sessionStart` signal: ms timestamp captured at window mount
   (key change). Marker injection requires `cursor < server_time
   <= sessionStart` — anything arriving DURING focus session is
   live-read by definition, never spawns a new marker. Pre-existing
   markers (from before mount) survive intact.

Tests: TDD red → green. New Playwright e2e
`marker-target-window-regression.spec.ts` with two test cases mirroring
vjt's exact prod repro flow. Both failed pre-fix with literal evidence
(`Expected: 0, Received: 1` for marker count; dist > 1000 for scroll).
Both pass post-fix. 713/0 vitest, 1411/0 mix, scripts/check.sh EXIT=0.

### S1.3 — Own-nick DM filter leaks every inbound DM (commit `178bd5b`)

Symptom: vjt opened `grappa` (own-nick) query window post-deploy and
saw EVERY inbound DM (CristoBOT replies, prior peer conversations)
leaked into it.

Root cause: `Scrollback.fetch/5` → `channel_or_dm_where/2` applied
uniform `WHERE channel == ^name OR dm_with == ^name` filter. For
peer-DM windows this is correct (outbound at `channel = peer`,
inbound at `channel = own_nick AND dm_with = peer`). But for the
OWN-NICK window, the `channel == own_nick` clause matched every
inbound DM the server canonically stores at `channel = own_nick,
dm_with = peer`. CP14-B3 regression (commit 47866bc, 2026-05-07) —
latent for 3 days because cic's in-memory scrollback masked it
until S1.1 deploy invalidated the SW cache.

Fix: new `Scrollback.fetch/6` takes explicit `own_nick`. When `name
== own_nick` (case-insensitive — IRC nicks are CI), filter restricts
to `channel == ^name AND dm_with == ^name` (self-msgs only).
Controller resolves own_nick via `Session.current_nick/2` (already
exists). `fetch/5` kept as nil-passing wrapper for back-compat.

Tests: TDD — 4 new cases in `scrollback_test.exs` describe "fetch/5
— :dm_with bidirectional DM": own-nick narrowing, peer DM unchanged,
case-insensitivity, fetch/5 wrapper preserves legacy OR. 1411/0
failures, scripts/check.sh EXIT=0.
