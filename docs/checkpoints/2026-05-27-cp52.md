## S1 — 2026-05-27 — PWA manifest id + bastille cic-only deploy path

vjt reported an Android user seeing "developed for an earlier version
of Android and lacks the latest privacy protections" at PWA install
time, on a phone that had **never opened cic before**. Root cause:
Google's WebAPK Minting Server caches minted APKs keyed on the
manifest hash. An APK minted months ago with a now-stale
`targetSdkVersion` was being served to every new Android installer
of cic, triggering Play Protect's blanket block. The user had inherited
a stale APK someone else's install had originally caused to be minted.

Fix: add `id: "/cic"` to the W3C manifest (`cicchetto/vite.config.ts`).
The `id` field is a same-origin URL, never fetched — pure primary key
used by browsers + WebAPK minter to answer "is this the same app?".
Adding it shifts the manifest hash, forcing the minter to mint a fresh
APK with current targetSdkVersion. Existing installs are unaffected
because Chrome reconciles via `start_url` ↔ `id`. The comment in
vite.config.ts spells out the format + the never-change-this-after-
first-install rule so future edits don't orphan the install base.

The deploy revealed a substrate bug. Initial `jail_deploy_cic.sh`
(new script — first cic-only deploy path on the bastille jail,
parallel to `scripts/deploy-cic.sh` for Docker) ran clean but the
broadcast endpoint returned 204 "no bundle on disk." Investigation
showed `Grappa.Cic.Bundle.@bundle_path` hardcodes
`runtime/cicchetto-dist/index.html` (Docker compile-time anchor where
compose.yaml bind-mounts the container's `/app/dist`), but jail's
vite emitted to `cicchetto/dist/`. nginx served the new bundle fine
(it has its own symlink) but the refresh banner broadcast was silent.

Right fix: align the jail to the Docker substrate. `jail_cic_build.sh`
now passes `--outDir ../runtime/cicchetto-dist` to vite; nginx symlink
repoints at the shared path; `deploy.sh` cold path collapses to call
`jail_cic_build.sh` (DRY). One server-side anchor for both substrates,
zero per-substrate special-casing in `Grappa.Cic.Bundle`.

Verified live on `irc.sindro.me`: `id: "/cic"` present in manifest,
healthcheck green, refresh banner broadcast (hash `BydkWNJw`) reached
all connected user-topics. Two commits (`47f4bf0`, `bd1d85c`) pushed.
**BEAM never touched** — entire fix landed without restarting the
bouncer or disturbing any IRC session. First end-to-end exercise of
the cic-only deploy path on bastille.

## S2 — 2026-05-27 — Bastille XFF regression: loopback peer + nginx-as-local-proxy

Spotted during cp52 S1 close: vjt asked about a session with a
loopback-shaped IP and queried whether XFF was broken. Live DB query
confirmed all post-bastille user sessions persisted `ip = "127.0.0.1"`
instead of real client IPs — silent audit-trail data loss since the
bastille jail went live.

Root cause: the `RemoteIpFromProxy` wrapper (cp51-era, built for the
Docker substrate where loopback = container-shell exclusively) bypassed
`RemoteIp.call/2` for ANY loopback peer. On bastille the shape inverts:
nginx runs in the same jail as grappa and proxies via 127.0.0.1:4000,
so every legitimate user request surfaces with `peer = 127.0.0.1` AND
nginx-set X-Forwarded-For. The bypass silently dropped the rewrite,
the persisted IP collapsed to loopback. Same trap latent in docker
prod (nginx + grappa on same host, grappa bound to loopback).

Fix: 3-row trust matrix in `RemoteIpFromProxy.call/2`:
- peer-loopback + no-XFF → trust peer (operator shell / healthcheck)
- peer-loopback + has-XFF → delegate to RemoteIp (local nginx proxy)
- non-loopback peer → delegate to RemoteIp (default chain walk)

Trade-off: shell-spoof of `X-Forwarded-For: 127.0.0.1` from a loopback
peer now passes `LoopbackOnly`. Explicitly accepted — anyone with
`sudo bastille cmd grappa` or `docker exec grappa` already has
root-equivalent access (BEAM kill, sqlite drop, codebase rewrite).
The defense at this layer is network reachability, not input
validation against an attacker with the keys. Documented in both
wrapper + LoopbackOnly moduledocs.

Deploy ended up COLD because the diff included `infra/freebsd/deploy.sh`
(refactored in cp52 S1's bd1d85c to call `jail_cic_build.sh` instead of
inline npm) — preflight rule at `preflight.ex:255` correctly forces
COLD on deploy.sh edits ("running an old version of the deploy script
after the new one landed"). Sessions reset ~5s. The hot-vs-cold
preflight gate worked as designed; the COLD was the cost of bundling
the wrapper fix with the deploy.sh refactor.

Tests: 13 wrapper-plug tests (refactored to assert the 3-row matrix;
loopback-no-XFF preserved as sanity, loopback-with-XFF added as the
previously-broken row). 2533 / 8 / 34 / 0 failures.

### Visitor session sharing — feature

Closes the 409 anon_collision gap: visitor on second device can't
log in via the normal nick path (would provision a fresh visitor
row → unique-violation). Sharing flow: mint signed token on device
A → consume on device B → both end up holding distinct
`accounts_sessions` rows pointing to the SAME `visitors.id`. Both
devices stay connected (sharing, NOT transfer).

8 buckets, all delivered:
- B1 (`730b3f3`) — `Grappa.Visitors.ShareTokens` GenServer + ETS
  one-shot `:ets.insert_new/2` + supervision before Endpoint
- B2+B3+B4 (`e95c985`) — `POST /me/share-token` (visitor-only,
  user→403) + `POST /auth/share/consume` (unauth; signed token IS
  credential) + telemetry events
- B5+B6 (`4ff695d`) — cic Settings "share session" button (visitor
  gate) + ShareSessionModal (URL + copy + countdown) +
  `/share/:token` plain SPA route + `installSharedSession()` helper
- B7 (`a0b5ed5`) — Playwright e2e (1.4s, 3 contexts: mint, consume,
  one-shot rejection)
- B8 (`c55f3cc`) — DESIGN_NOTES section + README checklist

Pre-deploy audit caught a separate gap: the slash-commands bundle
(`24eb1d8` — issues #20/22/23) had shipped with vitest + elixir
unit tests but NO Playwright e2e per `feedback_ux_e2e_mandatory`.
Backfilled in `8b70d5d` — 5 specs for /j auto-prefix,
context-aware /topic family, /q close, /quote PING liveness.

Storage decision: Phoenix.Token + supervised ETS (NOT a DB table)
— threat model benign (operator clicks own link twice), TTL 10min,

Wire shape: mint returns `{token, expires_at}` ISO8601;

atoms split deliberately — `share_token_expired` vs
`share_token_consumed` so cic copy + telemetry can distinguish.

### Deploy fight (3 incidents, 3 fixes)

The new supervised child (`Grappa.Visitors.ShareTokens` in
`application.ex`) correctly triggered COLD via preflight. The COLD
path then exposed THREE pre-existing layers of fragility in the
bastille deploy pipeline that the prior memory
`[bastille_deploy_pipeline_hardened]` claimed were closed:

1. **Async-stop race** (`d8f354c`). `service grappa stop` returns
   immediately while BEAM drains over 3-10s. `pkill epmd` during
   that window triggers BEAM to respawn epmd; new `service grappa
   start` races against the still-registered `grappa@grappa` node
   name → `Protocol 'inet_tcp': name in use` → new BEAM crashes →
   deploy fails at healthcheck. Fix: poll `pgrep beam.smp` until
   old BEAM gone (20s timeout + SIGKILL fallback) THEN pkill epmd.

2. **Self-modifying script trap** (`55f0415`). `/bin/sh` read-ahead
   buffers the script file. When `deploy.sh` git-pulls a new
   version of itself, downstream steps continue executing the
   PRE-PULL bytes — fix takes effect on the SECOND deploy, not the
   first. Silent (mix release runs, supposedly-fixed stop+start
   block executes old logic). Caught on the second incident:
   d8f354c's wait-loop fix didn't run because the running deploy.sh
   was still the pre-d8f354c version. Fix: `cmp` running script vs
   on-disk post-pull; `exec` new version with `DEPLOY_REEXECED=1`
   guard against infinite loop.

3. **Preflight rule was wrong** (`8e05cd5`). `infra/freebsd/deploy.sh`
   was classified `:image_substrate` → COLD ("running an old
   version risks divergent behavior"). Wrong: shell scripts don't
   bake into the BEAM, the rc.d daemon, or the next-spawn env.
   COLD-restart for a deploy.sh edit is 30s of pointless downtime
   — the new bytes are on disk for the next deploy regardless of
   how this one classifies. Asymmetric too: `scripts/deploy.sh`
   (Docker) was already HOT. Removed the rule; now both deploy
   orchestrators classify HOT. The rule itself was the bug that
   forced the COLD path which triggered the race.

Recovery for both incidents: inside the jail,
`kill $(pgrep epmd) && service grappa start`. Worked because by
then the old BEAM had long since exited; only epmd's cached
`grappa@grappa` registration was wrong.

### Latent bug fix (separate from the deploy fight)

`6b2979d` — `Session.Server.handle_call({:send_join, ...})` had a
narrow case clause `:ok | {:error, :invalid_line}`. The third
return shape from `Client.send_join` — `{:error, :no_socket}` —
crashed with CaseClauseError when vjt's `/join #sniffo` hit a
Session whose upstream TLS was failing
(`:unsupported_record_type 58`). The crash bubbled as HTTP 500.

Widened three sites symmetrically (send_join handle_call,
send_part handle_cast, autojoin reduce after 001 RPL_WELCOME) to
match the pattern every OTHER `Client.send_*` call site already
used: `{:error, _} = err -> {:reply, err, state}`. send_join now
surfaces `{:error, :not_connected}` → FallbackController emits
clean 400 instead of 500. Session stays alive; next backoff tick
retries the socket.

Root pattern: 3-of-N `Client.send_*` handlers in `server.ex` had
narrow case clauses; the rest used the wide `{:error, _}` form.
These three were outliers that survived because the dead-socket
path is rare in CI but inevitable in prod under upstream churn.
