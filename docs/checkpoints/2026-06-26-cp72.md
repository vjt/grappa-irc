## Exact implementation (≈3 lines + 1 test)

**File:** `cicchetto/src/lib/slashCommands.ts`, the `msg:` parser (currently
~lines 251-261):

```js
msg: (verb, rest) => {
  const sp = rest.search(/\s/);
  if (sp === -1 || sp === 0) {
    if (rest === "") return err(verb, "/msg requires <nick> <text>");
    return err(verb, "/msg requires message text after nick");
  }
  const target = rest.slice(0, sp);
  const body = rest.slice(sp + 1).trim();
  if (body === "") return err(verb, "/msg requires message text after nick");
  // #12 — /msg is for nicks (queries); grappa does not relay PRIVMSG to a
  // channel addressed by name (even one that accepts external msgs). Reject
  // up front so no phantom query window keyed by a channel name is opened.
  if (["#", "&", "!", "+"].includes(target[0] ?? "")) {
    return err(verb, "/msg to a channel is not supported");
  }
  return { kind: "msg", target, body };
},
```

**Why this fixes the phantom window:** `err()` returns an error-kind
`SlashCommand`; `compose.ts`'s executor short-circuits error commands BEFORE
the `case "msg"` block (which is what calls `openQueryWindowState` +
`setSelectedChannel` at compose.ts ~335-336). Pattern precedent:
`compose.ts:189` handles `kind === "empty"` early. VERIFY that error-kind
short-circuit before writing (grep compose.ts for how the parsed cmd's error
kind is turned into `{error}` — the existing /msg validations already surface
inline, so this just adds one more).

**Services shortcuts are unaffected:** `/ns`/`/cs`/etc. rewrite to
`{kind:"msg", target:<ServiceNick>}` via a SEPARATE code path
(slashCommands.ts ~392-445), not through this `msg:` parser, and their
targets are service nicks (never `#`-prefixed).

**Channel sigils:** `#` `&` `!` `+` (0x23/0x26/0x21/0x2B) — see
`channelKey.ts:canonicalChannel`. No shared `isChannel` predicate exists
(only an inline byte check in `pushPayload.ts:102`); inline the 4-char check.

## #12 root cause (reference — why the current behavior is broken)

`compose.ts` `case "msg"` routes `/msg #X` through the QUERY path:
`openQueryWindowState(networkId, "#x", …)` + `setSelectedChannel(kind:"query")`
— a query window keyed by a CHANNEL name. cic's own-send render is WS-driven
on the per-channel topic (`scrollback.ts:235-240`, no optimistic append), and
cic only subscribes to channel topics for JOINED channels — so a "#x" query
window hears nothing and the message never renders (live AND post-restore).
The reject sidesteps the whole semantic mismatch. (Server `Session.send_privmsg`
currently HAPPILY sends to a non-joined channel + persists to scrollback("#x"),
which is why it later shows in archive — left as-is; the cic reject means cic
never sends it.)
